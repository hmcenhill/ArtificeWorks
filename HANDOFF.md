  - **5.1 model + seed.** New domain: `Component` (fungible input material with an `OnHand` count — distinct from `Product`, the finished automaton, and from `StockKeepingUnit`, a serialized finished unit), `BomLine` (flat BOM entry on `Product`), `ComponentDemand`, `MaterialReservation`/`MaterialReservationLine`. `Product.ComputeDemand(qty)` expands the BOM (`QtyPerUnit × OrderItemQty`) **ordered by component id** — that ordering is load-bearing for deadlock avoidance, not cosmetic. Migration `BomAndComponentInventory` (4 tables, unique index on `material_reservations.WorkOrderId`, unique `(ProductId, component_id)` on BOM lines, `CHECK (on_hand >= 0)`). `CatalogSeeder` (idempotent, additive, runs at API startup when no migrations are pending) seeds 7 shared platform components + 3 trade-specific per line — **every product is exactly 70% shared, asserted in a test, not just claimed**. `GET /products/{id}` now returns the BOM with per-component on-hand.
  - **5.2 picking consumer.** `Application/Materials/MaterialPickingService` owns the workflow (loads order → dedupe pre-check → expand BOM → reserve → note history → publish); the worker's `WorkOrderScheduledHandler` is now a thin adapter that adopts the inbound correlation id and logs the outcome. The 4.2 "scheduling acknowledged" stand-in is gone. New event `MaterialsReserved` (`work-order.materials-reserved`). **Picking deliberately does not advance Scheduled → InProcess** — that's Epic 6's transition; picking hands over by publishing. Workers now uses `AddRabbitMqMessaging` (it publishes as well as consumes).
  - **5.3 all-or-nothing + concurrency.** `MaterialReservationRepository.TryReserve` runs one transaction: per line, an **atomic conditional decrement** (`UPDATE components SET on_hand = on_hand - @q WHERE "ComponentId" = @id AND on_hand >= @q`, checking rows-affected); zero rows → roll the whole thing back, so a partial BOM leaves on-hand exactly as it was. No aggregate rowversion needed (consistent with the standing no-optimistic-concurrency decision). Shortage → `SetHold` naming the short component, reason in state history, **message acked** (business outcome, not a fault). Proven by a real 10-way race against stock for 4: exactly 4 reserve, 6 go OnHold, `sum(reserved) == on_hand`.
  - **5.4 idempotency.** The reservation row **is** the dedupe key (unique index on `WorkOrderId`), so the marker and the reservation commit atomically by construction — no inbox table. A cheap pre-check handles the common case; the constraint handles the race (loser's `DbUpdateException` 23505 → rollback takes its decrements with it). Duplicates ack, are logged, and deliberately write **no** history note (a note per redelivery would itself be non-idempotent). Both sequential and simultaneous redelivery are tested.
  - **Testing shape worth knowing:** the reservation guarantees are raced in `MaterialPickingTests` (Postgres only, workflow invoked in parallel scopes) because the broker path *can't* prove them — prefetch 1 on a single consumer serializes deliveries. `WorkerConsumerTests` (Postgres + RabbitMQ) owns the plumbing claim end-to-end. Epic 6 reuses this split exactly.
- **Epic 6 (production + inspection) complete** — 6.1–6.4 in one pass. **The pipeline now runs unattended end to end:** one `POST /work-orders/{id}/advance` to schedule, and events alone carry the order Scheduled → InProcess → Inspection → Delivery, with a bounded, visible, recoverable failure loop in the middle. **149 tests green (106 unit + 43 integration).**
  - **6.1 units + production.** `StockKeepingUnit` stopped being decorative: it has a `UnitStatus` (Built/Passed/Scrapped), `BuiltUtc`/`InspectedUtc`/`ScrapReason`, and the `BuildAttempt` that made it. Work-order↔SKU went **many-to-many → one-to-many** with an owning `work_order_id` FK. New `WorkOrder.Build(by, attempt)` creates the *outstanding* quantity and advances Scheduled → InProcess; `AssignSku`'s fulfilment guard now counts **non-scrapped** units (`LiveQty`), which is what makes rebuilding a shortfall possible at all. `ProductionService` + `MaterialsReservedHandler` bind `work-order.materials-reserved` — the key Epic 5 had been publishing to nobody.
  - **6.2 inspection + verdicts.** `InspectionService` advances InProcess → Inspection and verdicts each unit of the attempt via `IVerdictSource` (`RandomVerdictSource`, `Inspection:FailureRate` default `0.0`, optional `Seed`). `POST /work-orders/{id}/inspections` records a verdict by hand with stable ProblemDetails codes (`unit_not_found`, `order_not_in_inspection`, `unit_already_inspected`, `scrap_reason_required`). **Both paths share one `Resolve`**, so a factory run by a visitor and one run unattended reach identical state. New `Inspection:AutoInspect` (default true) — turn it off and units wait in Inspection for a human, which is what makes the endpoint reachable at all when the failure rate is 0. Units and verdicts are on the work-order read model.
  - **6.3 rework + cap.** `WorkOrder.ReturnToProduction` is the state machine's first backward transition (Inspection → InProcess only). New guarded **`WorkOrder.Fault(by, reason)`** rather than reaching for `SetStatus` from a worker — the story flagged that as the signal, and it was. `Production:MaxRebuildAttempts` (default 3) counts *attempts*, not scrapped units: attempts 1–4 run, a shortfall after attempt 4 would be rebuild #4 and faults instead. Rework is a real cycle over the broker — the worker publishes `rework-required` and consumes it.
  - **6.4 idempotency.** Epic 5's "the record is the key" survives, but attempt-scoped: new `production_runs` and `inspection_runs`, each **unique on `(WorkOrderId, AttemptNumber)`**, each written in the *same* `SaveChanges` as the work it describes — so a losing duplicate's units and transition roll back with its failed insert. The attempt number is **derived from the event** (materials-reserved ⇒ 1; rework of N ⇒ N+1), never read from current state, so a redelivery computes the same key and collides. `inspection_runs` is not redundant with the per-unit guard: without it a redelivered `ProductionCompleted` could publish a second `InspectionPassed` or burn a second rebuild attempt.
  - Migration `ProductionAndInspection`: SKU columns + owning FK (drops `work_order_skus`), `work_orders.BuildAttempt`, the two run tables, and **the long-outstanding `CompletedBy` mapping** — read-only properties are only mapped when configured explicitly, which is why it was silently missing since 4.2. It now round-trips, asserted in an integration test.

## Open decisions

- `SetStatus` (superuser override) still has no endpoint — deferred to admin auth, as planned.
- **No optimistic concurrency on `WorkOrder`** (no rowversion / `xmin`). Concurrent *same-order* advances would duplicate history / lose updates silently — so 3.4 tests concurrent *create* only. Deferred; the Epic 4 async pivot moves transitions onto a queue and will revisit aggregate concurrency. (Detailed in `docs/Plan/.../3.4.md`.)
- **Publishing is best-effort, at-most-once** (4.1, and now for every Epic 6 event too). Publish-after-commit with a swallow-and-log on broker failure: a state change can persist while its event is dropped (the classic dual-write gap). Chosen to keep the core resilient to broker outages. Epic 8's transactional outbox closes it. **This now matters more**: a dropped `ProductionCompleted` or `ReworkRequired` stalls an order mid-pipeline with no retry, where before it only stalled picking.
- **Releasing a held work order does not re-trigger picking** (5.3), and **releasing a Faulted order does not re-trigger anything either** (6.3). Release only restores the previous status and publishes nothing, so a released Fault sits in Inspection with every unit already judged — recoverable by hand, not automatically. Deliberate and tested; the simulation engine (Epic 10) is the natural place to close both loops.
- **Rebuilds consume no new materials** (decided at 6.3, option 1 of three). Physically a scrapped unit burns its parts, but 5.4 made `material_reservations.WorkOrderId` unique, so a second pick per order is impossible without reopening that design. The original pick covers the order and scrapped parts are notionally salvaged. **Known simplification**, revisited in Epic 13 with multi-level BOMs; the honest fix is widening the reservation key to `(WorkOrderId, AttemptNumber)` to match the run tables.
- **Epic 5's idempotency trick did not generalise, and that is recorded in three places** — `ProductionRun`'s doc comment, `docs/messaging-topology.md`, and here — because it looks like a regression later if the reasoning isn't written down. Order-scoped ⇒ attempt-scoped is the whole of 6.4.
- **The work order still has no optimistic concurrency**, and Epic 6 leans on the run tables' unique keys instead. That covers duplicate *deliveries of the same attempt*, which is the realistic case. It does not cover a manual API verdict racing the auto-inspector's `SaveChanges` on the same order: the unit-level guard resolves who wins the verdict, but a genuinely simultaneous manual verdict could lose its history note to a last-write-wins update. Narrow enough to accept; revisit if the aggregate ever gets a rowversion.
- **The pick's state-history note is written outside the reservation transaction** (5.2). The decrements + reservation row commit atomically (that's the invariant); the note is a second save. A crash in between leaves a correct reservation with no note, and the redelivery would be a no-op duplicate — so the audit trail can lose a line the inventory never does. Deliberate: keeping audit out of the hot transaction. Revisit if the dashboard ends up depending on the note.
