- **Epic 8 (reliability + recovery) complete** — 8.1–8.4 in one pass. **Nothing gets through the pipeline by luck any more:** every publisher writes its event in the same transaction as the work, a failed handling climbs a retry ladder instead of being dropped, whatever genuinely can't be handled becomes a row a human can read and replay, and a client retry can't double-create an order. **210 tests green (116 unit + 94 integration).** Migration `ReliabilityAndRecovery` (3 tables; the `xmin` mapping needs no DDL).
  - **8.1 the outbox, and the concurrency token.** `IEventPublisher` now resolves to `OutboxEventPublisher`, which `Add`s an `outbox_messages` row to the caller's `DbContext` **and does not save** — so it is flushed by whatever `SaveChanges` commits the work. **Every publish site was reordered from publish-after-commit to stage-before-commit**; that is the whole change, and the rule for new ones is one line: *stage the event before the save, never after it.* `PublishSafely` is gone from all four services. `OutboxDispatcher` (both hosts, ~1s poll) claims in id order with `FOR UPDATE SKIP LOCKED`. Publishing is now **at-least-once on purpose** — the dispatcher can die between publishing and marking sent — which is only safe because Epics 5–7 built a dedupe key at every stage. **`WorkOrder` finally has `xmin` as a concurrency token**, mapped as a shadow property (`UseXminAsConcurrencyToken` was removed in Npgsql 9; it's an explicit `Property<uint>("xmin")` now).
  - **8.2 the retry ladder.** Epic 4's `nack(requeue: false)` — which *dropped* every failed message — is replaced by a three-way classification in the consumer loop: business outcome → ack, transient → next rung, poison → park. **Three fanout exchanges, not one direct `artifice.retry`**: a delay queue dead-letters on expiry under the message's *own* routing key, so the rung has to be encoded in the exchange or the original event type is lost. Three retries, four deliveries. Attempt count in an `x-attempt` header we control rather than parsing `x-death`. An unknown routing key is **poison, not a no-op** — silently acking it away is how a message disappears.
  - **8.3 dead letters.** `ParkedQueueDrain` (a *separate* consumer — the pipeline's handlers must never run for a parked message) turns `artifice.parked` into `dead_letters` rows. `GET/POST /system/dead-letters…` — under `/system` so the admin-auth gate lands on a path prefix. Replay goes **through the outbox** and resets the ladder to attempt 1; a second replay needs `?force=true`. The drain is the most defensive code in the system: an unparseable body still gets a row, and only a *database* failure makes it requeue.
  - **8.4 idempotency keys.** `Idempotency-Key` on `POST /work-orders` via an `[Idempotent]` resource filter. The key row is **staged before the action and committed by the action's own `SaveChanges`** — so the work order, its outbox row and the key are one transaction, which is the whole epic in a single commit. The unique index is the guarantee, not the pre-check: the loser of a race rolls back entirely and replays the winner's stored `201`. Same key + different body → 422 `idempotency_key_reused`; claimed-but-unfinished → 409 `idempotency_key_in_flight`. No header → unchanged, proven by the untouched existing suite.
  - **A footgun worth remembering:** `Retry:DelaysMs` in appsettings would have *appended* to a non-empty default (the config binder binds array indices into the existing array), silently producing a six-rung ladder. `RetryConfiguration.DelaysMs` therefore starts empty with the shipped default behind it in `Delays`.

## Open decisions

- `SetStatus` (superuser override) still has no endpoint — deferred to admin auth, as planned. **`/system/dead-letters` now waits behind the same gate**, which is why it lives under a path prefix: the eventual auth boundary falls on one route rather than a scattering of endpoints.
- **Publishing is at-least-once, deliberately** (8.1, replacing 4.1's at-most-once). The outbox dispatcher can die between `BasicPublish` and marking the row sent, so an event can go out twice. That is the correct trade — a duplicate is answered by the dedupe keys 5.4/6.4/7.1/8.4 established, a loss was answered by nothing. **If a future stage needs a new guard to survive a duplicate, that is a finding about its key, not a reason to weaken the outbox.**
- **Ordering is per-host, not global** (8.1). Rows are claimed and published in id order, so one host's events leave in the order it wrote them; the API's dispatcher and the worker's are not coordinated. Benign today because consecutive events for one work order are separated by an actual delivery and a handler, so one host writes them at a time. Revisit if a host is ever scaled out *and* a stage starts publishing two events for one order.
- **Release re-triggers exactly one thing, and only at Delivery** (7.3). A release that lands in Delivery *with no shipment* republishes `InspectionPassed`; every other release is inert. Both halves are tested. **Closed by Epic 10**: simulated actors create orders and run stages but **never release a hold** — holds stay a visitor's decision and 10.4's sweep retires whatever nobody rescues — so "refusal is uncapped" needs no cap. Re-open only if Epic 12 adds an automatic recovery action.
- **Pacing is quantized, and applied only in the outbox dispatcher** (10.1). A stage's duration snaps to the nearest rung of a fixed ladder, and jitter chooses *which* rung — small edits to a duration may move nothing, and a message already in a delay queue keeps its old timing. Both are correct; `PUT /system/simulation` returns the resolved rung so neither looks like a bug. Exact per-message delay would mean the community delayed-message-exchange plugin and a custom RabbitMQ image to self-host in M7, for precision nobody can perceive. **The pace ladder is declared on connect by every host**, not just the consumer, because the dispatcher runs in all three and publishing to a missing exchange closes the channel.
- **The dials are eventually consistent across hosts** (10.2). A `PUT` reaches the worker within one refresh interval (~5s), not instantly; the response says when it lands. Making it instant means a query per decision or a broadcast, both worse than a few seconds' lag on a demo dial. The row is the authority and appsettings is only its default; **seeds stay in configuration** (a live-editable seed is a flake waiting to happen), and this is **global tuning, not per-order failure injection** — that is Epic 12, with a different guard and blast radius.
- **The world reset is a sweep, not a truncate** (10.4). It restocks components to `seed_on_hand` and retires old terminal/held/faulted orders, and **refuses to touch anything in flight, the catalog, or `dead_letters`** — which is what makes "without downtime" fall out for free. It does **not** reset the settings row (that is the floor, not the dials someone turned). Restock is a conditional set that only ever raises stock, so it stays idempotent and can never drop below a live reservation's claim; its updates are ordered by component id, the same anti-deadlock rule 5.2's picking follows, because it is the second bulk writer against `components`.
- **`WorkOrder.Origin` is a real column, and a two-valued metric dimension** (10.3). Order id/correlation id/trace id stay off metrics (9.2's rule), but `Visitor`/`Simulated` is exactly the bounded label metrics *do* take — without it `/system/stats` and every panel report robot traffic as demand. The generator creates orders **over HTTP**, not by a direct write, deliberately: a direct write is the shortcut the epic's own note warns about, and would skip 8.4's idempotency, the DTO validation and the outbox row.
- **Rebuilds consume no new materials** (decided at 6.3, option 1 of three). Physically a scrapped unit burns its parts, but 5.4 made `material_reservations.WorkOrderId` unique, so a second pick per order is impossible without reopening that design. **Known simplification**, revisited in Epic 13 with multi-level BOMs; the honest fix is widening the reservation key to `(WorkOrderId, AttemptNumber)` to match the run tables.
- **Epic 5's idempotency trick did not generalise, and that is recorded in three places** — `ProductionRun`'s doc comment, `docs/messaging-topology.md`, and here — because it looks like a regression later if the reasoning isn't written down. Order-scoped ⇒ attempt-scoped is the whole of 6.4; 7.1 went back to order-scoped and 8.4 went to the client's request, all under one rule: **the key follows the thing that must happen once**.
- **Telemetry is never load-bearing, and that is enforced by defaults** (Epic 9). `Telemetry:OtlpEndpoint` is empty in the shipped `appsettings.json` — instruments and activities are real, nothing leaves the process — and Development sets the compose stack's endpoint. No collector reachable is a startup warning; a blocked exporter has a 5s ceiling; the worker's health listener failing to bind is a warning, not a stop. **The pipeline's Epic 8 guarantees must never acquire a dependency on the thing that watches them.**
- **A metric never gets a per-order label** (9.2). Order id, correlation id and trace id are unbounded cardinality. Those questions belong to traces and logs, and that division is the argument the whole epic makes. The gauges read a cached snapshot rather than querying — **no metric collection issues a database query**, or a Grafana refresh interval becomes a load generator.
- **`ArtificeWorksMetrics` and `ArtificeWorksTelemetry` live in Application, not Infrastructure** — a deliberate deviation from 9.2's wording. The workflow services that record them are in Application and it cannot reference Infrastructure; Domain would be lower but depends on nothing, including `System.Diagnostics`. The OTel *registration* is in Infrastructure as the story intended.
- **The retry ladder is fixed at three rungs and never gives up quietly.** A message that exhausts it parks forever — no TTL, no expiry — and only a human replaying it moves it. That is intentional (silence is the failure mode this epic removed) but it does mean `dead_letters` grows unboundedly if nobody looks; the retention sweep only removes rows that *have* been replayed.
