## Visitor affordances: create an order, make the decisions, turn the dials

**Labels:** frontend, demo

## Summary

Give the visitor hands. Create a work order from a product template, make the decisions the hybrid
model leaves to a human — approve the schedule (advance), choose a carrier, put an order on hold and
release it, record an inspection verdict — and turn the factory's live dials through the settings
endpoint. Almost entirely frontend: every endpoint this needs already exists and was built to be
called from here.

## Why

The epic's headline criterion is "a visitor can create a work order and watch it travel the whole
pipeline," and until this story the visitor can only watch. The board (11.1) and the live feed
(11.2) make the factory legible; this makes it *interactive*, which is the difference between a
screensaver and a demo a hiring manager can poke.

The decision moments are the portfolio point made concrete. The system was deliberately built so a
visitor's action and a simulated actor's action run the **same** path — 6.2's verdict endpoint and
7.x's booking endpoint each say so in their own docstrings. This story is where that design pays
off: the UI calls the ordinary endpoints, and a visitor-driven order is indistinguishable in state
from a robot-driven one, only tagged `Visitor` in origin. And the simulation dials (10.2's
`PUT /system/simulation`) are the "watch this" button — turn the failure rate up and the next order
starts failing inspections live, no restart, exactly what that story built the row to allow.

## Tasks

- [ ] **Create from a template:** a form that reads `GET /products` for the catalog (the three
      lines — Custodian, Delver, Courier), picks a product and quantity, and `POST /work-orders`
      with an `Idempotency-Key` (8.4) so a double-submit is one order. On success, route to that
      order's live detail view so the visitor watches the thing they just made — the `origin=Visitor`
      filter from 11.1 makes "find mine" trivial
- [ ] **The decision moments,** surfaced on the order detail only when the order's state allows the
      action (the endpoints enforce it with 409s; the UI should not offer a move that will bounce):
      - approve/advance — `POST /work-orders/{id}/advance`
      - choose a carrier — `POST /work-orders/{id}/shipments` (offered in Delivery when auto-book is
        off), surfacing `unknown_carrier` / `carrier_unavailable` distinctly
      - hold / release — `POST …/hold`, `POST …/release` (the visitor is the *only* thing that
        rescues a held order — 10.x's sweep retires whatever nobody rescues)
      - record a verdict — `POST …/inspections` for a serialized unit (the manual inspector; also
        the door Epic 12 will use)
- [ ] **Map ProblemDetails to human text.** The API speaks RFC 7807 with stable codes (3.3); a
      shared error component turns `order_not_in_delivery`, `shipment_already_booked`,
      `idempotency_key_reused` et al. into a sentence and a state, not a raw 409. One mapping table,
      keyed by problem `code`
- [ ] **The dials:** a controls panel reading `GET /system/simulation` and writing
      `PUT /system/simulation` — failure rate, refusal rate, pacing, generation on/off. Show the
      response's *source* (`configured` / `overridden`) and the **resolved rung** for a pacing edit
      (10.2's note: a small duration change may not move a rung, and that looks like a bug unless the
      UI shows the rung it landed on). Reject out-of-range in the UI too, but let the 422
      (`simulation_setting_out_of_range`) be the authority
- [ ] Make it clear the dials are **global** — they retune the whole factory, not one order. That is
      10.2's line in the sand against Epic 12; the panel should not imply per-order targeting that
      doesn't exist yet
- [ ] Phone-legible forms and controls (the standing criterion)

## Acceptance Criteria

- [ ] A visitor creates a work order from a template and is taken to its live timeline
- [ ] The visitor can advance, book a carrier, hold/release and record a verdict where the order's
      state allows it, and each rejection is shown as readable text with the reason
- [ ] A visitor-created order is tagged `Visitor` and travels the identical pipeline a simulated
      order does
- [ ] The simulation dials change live factory behaviour and the panel reflects the resolved rung
      and the setting's source
- [ ] A double-submitted create makes one order, not two

## Decisions (to confirm at story start)

- **The UI calls the ordinary endpoints; there is no dashboard back door.** The whole point of the
  same-path design is that a visitor action *is* a pipeline action. A shortcut here would be the
  smell the epic warns about.
- **Offer only state-legal actions, but trust the API as the authority.** Client-side gating is UX,
  not enforcement; the 409/422 contract is the real guard and drives the message when a race loses.
- **The dials are global tuning, not failure injection.** Per-order "fail *this* one" is Epic 12,
  with a different blast radius. This panel is 10.2's global knobs and must not pretend otherwise.
- **Create sends an `Idempotency-Key`.** The endpoint offers it precisely for a fat-fingered
  double-click; using it is free correctness.

## Notes

Depends on 11.1 (create routes into the detail view) and reads best with 11.2 live (the created
order animates immediately). No new backend is expected — if a decision moment seems to *need* a new
endpoint, that is a finding worth raising, because the endpoints were built to be driven from here
and a gap means one of them isn't as reusable as its docstring claims.

Leaves `SetStatus` (the superuser override) alone: it still has no endpoint and sits behind the
deferred admin gate, and `/system/simulation` being unauthenticated is the same known deferral —
fine for the demo, flagged in architecture.md.
