## Lifecycle command endpoints

**Labels:** api, backend
**Milestone:** M2

## Summary

Expose the domain's existing lifecycle commands over REST: `POST /work-orders/{id}/advance`, `POST /work-orders/{id}/hold`, `POST /work-orders/{id}/release`.

## Why

The domain already enforces these transitions (`AdvanceToNextStep`, `SetHold`, `ReleaseHold`); the API just can't reach them yet. These are also the buttons the dashboard will press.

## Tasks

- [x] Request shape carrying actor (`createdBy`) and optional notes — `WorkOrderCommandRequest`
- [x] Handler methods invoking the domain commands and persisting — `WorkOrderHandler.Advance/Hold/ReleaseWorkOrder`
- [x] Rejected transitions surface *why* — domain now returns `TransitionResult`; API returns 409 with the reason (3.3 will make it ProblemDetails)
- [x] Decide whether `SetStatus` (superuser override) gets an endpoint now or waits for admin auth — **waits** for admin auth

## Acceptance Criteria

- [x] Valid advance/hold/release calls transition the order and append state history
- [x] Invalid calls are rejected with an informative error, not a silent `false`
- [x] Unit and integration tests cover both outcomes

## Notes

The domain methods currently return `bool` with comment-only error reasons — this story will likely motivate returning richer results from the domain. That's a worthwhile refactor, not scope creep.
