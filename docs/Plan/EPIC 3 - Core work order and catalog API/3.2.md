## Cancellation

**Labels:** api, domain
**Milestone:** M2

## Summary

Add a `Cancelled` terminal state to the domain and expose `POST /work-orders/{id}/cancel`.

## Why

Cancellation demonstrates business rules and non-happy-path lifecycle control — and real factories cancel work orders constantly.

## Tasks

- [x] Add `Cancelled` to `WorkOrderStatus` and define allowed cancellation states (proposal: any non-terminal state; cancelling InProcess or later should release/return assigned stock)
- [x] Domain command with guard + state history entry recording actor and reason
- [x] API endpoint and handler flow
- [x] Unit and integration tests, including each disallowed state

## Acceptance Criteria

- [x] Cancellation works in valid states and is terminal
- [x] Invalid cancellation attempts are rejected with reasons
- [x] Assigned stock handling on cancellation is defined and tested
- [x] Status history records the cancellation

## Notes

Document intentionally unsupported cancellation states in the endpoint's docs. The stock-release rule is the interesting decision here — write down the reasoning.

### Decisions made (3.2)

- **Terminal (non-cancellable) states = `Completed` and `Cancelled`.** `Fault` is treated as *non*-terminal and stays cancellable: it is a stuck error state, not a business-complete one, so cancel is the escape hatch out of Fault until real fault remediation lands in a later epic. Introduced a `_terminalStatuses` set on `WorkOrder` and reused it to generalise the terminal guards in `AdvanceToNextStep` and `SetHold`.
- **Stock release on cancel:** there is no inventory/warehouse aggregate yet, so "returning stock" means detaching the serialized units from the order (`_assignedStock.Clear()`), freeing them for reassignment. A cancelled order holds no physical stock. This runs on every cancel regardless of state (stock can currently be assigned in any state), which is a superset of the "InProcess or later" proposal and simpler to reason about.
- **Latent fix folded in:** the new terminal guard in `SetHold` also closes a pre-existing gap where a `Completed` order could be put OnHold.
- **No migration needed:** `CurrentStatus`/`Status` persist via `HasConversion<string>()`, so the new enum value is just a new string in an existing column.
