## Cancellation

**Labels:** api, domain
**Milestone:** M2

## Summary

Add a `Cancelled` terminal state to the domain and expose `POST /work-orders/{id}/cancel`.

## Why

Cancellation demonstrates business rules and non-happy-path lifecycle control — and real factories cancel work orders constantly.

## Tasks

- [ ] Add `Cancelled` to `WorkOrderStatus` and define allowed cancellation states (proposal: any non-terminal state; cancelling InProcess or later should release/return assigned stock)
- [ ] Domain command with guard + state history entry recording actor and reason
- [ ] API endpoint and handler flow
- [ ] Unit and integration tests, including each disallowed state

## Acceptance Criteria

- [ ] Cancellation works in valid states and is terminal
- [ ] Invalid cancellation attempts are rejected with reasons
- [ ] Assigned stock handling on cancellation is defined and tested
- [ ] Status history records the cancellation

## Notes

Document intentionally unsupported cancellation states in the endpoint's docs. The stock-release rule is the interesting decision here — write down the reasoning.
