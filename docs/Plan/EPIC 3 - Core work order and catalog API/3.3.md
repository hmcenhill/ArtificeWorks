## Error contract and API polish

**Labels:** api
**Milestone:** M2

## Summary

One consistent error shape (RFC 7807 ProblemDetails) across all endpoints, replacing ad-hoc `BadRequest(string)` responses.

## Why

The dashboard, integration tests, and any API consumer need errors they can rely on. Establishing the contract now is cheap; retrofitting it across the async epics later is not.

## Tasks

- [x] Adopt ProblemDetails for all 4xx responses, with a stable machine-readable reason code per failure type
- [x] Map domain rejections (invalid transition, bad request data, not found) onto the contract
- [x] Validation of request DTOs (e.g., qty > 0 fails at the API boundary, not as a 500 from the domain exception)
- [x] Swagger reflects the error responses

## Acceptance Criteria

- [x] Every non-2xx response is a ProblemDetails payload with a reason code
- [x] The same failure always produces the same shape and code
- [x] Integration tests assert on reason codes, not error strings

## Implementation notes

**Reason codes (the `code` extension on every ProblemDetails):** `validation_failed`,
`work_order_not_found`, `product_not_found`, `product_already_exists`, `terminal_state`,
`must_release_first`, `already_held`, `not_held`, `internal_error`. Defined in
`ArtificeWorks.Api/Errors/ProblemCodes.cs` â€” these strings are the public contract.

**Layering:** the domain owns *why* a transition failed. `TransitionResult` now carries a
`TransitionErrorCode` enum (TerminalState / MustReleaseFirst / AlreadyHeld / NotHeld)
alongside the human message; `WorkOrderCommandResponse` and the create responses carry
outcome discriminators up so the controllers map to codes without string-sniffing.
`ProblemCodes.ForTransition(...)` translates the domain enum to the wire code.

**Plumbing:** `ApiControllerBase.Problem(status, code, detail)` builds the payload via the
framework `ProblemDetailsFactory` and stamps the `code` extension. Model-validation 400s
(`[ApiController]` auto-validation, e.g. `qty >= 1` via `[Range]`, `[Required]` fields) are
routed through `InvalidModelStateResponseFactory` to the same shape with `validation_failed`.
`AddProblemDetails()` + `UseExceptionHandler()` make unhandled 500s ProblemDetails too
(`internal_error`). `[ProducesResponseType]` attributes surface the error shapes in Swagger.

**Status choices:** work-order create with a missing product â†’ 400 `product_not_found`
(the body is invalid, distinct from a missing work order â†’ 404). Duplicate product create
is now 409 `product_already_exists` (was 400) â€” a conflicting duplicate is the RFC-correct
409. Transition rejections stay 409 with their specific code.
