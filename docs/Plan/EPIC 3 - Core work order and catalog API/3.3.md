## Error contract and API polish

**Labels:** api
**Milestone:** M2

## Summary

One consistent error shape (RFC 7807 ProblemDetails) across all endpoints, replacing ad-hoc `BadRequest(string)` responses.

## Why

The dashboard, integration tests, and any API consumer need errors they can rely on. Establishing the contract now is cheap; retrofitting it across the async epics later is not.

## Tasks

- [ ] Adopt ProblemDetails for all 4xx responses, with a stable machine-readable reason code per failure type
- [ ] Map domain rejections (invalid transition, bad request data, not found) onto the contract
- [ ] Validation of request DTOs (e.g., qty > 0 fails at the API boundary, not as a 500 from the domain exception)
- [ ] Swagger reflects the error responses

## Acceptance Criteria

- [ ] Every non-2xx response is a ProblemDetails payload with a reason code
- [ ] The same failure always produces the same shape and code
- [ ] Integration tests assert on reason codes, not error strings
