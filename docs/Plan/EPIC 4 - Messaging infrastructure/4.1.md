## Event contracts and publisher

**Labels:** messaging, backend
**Milestone:** M3

## Summary

Define the first typed event contracts and a publisher abstraction, with the RabbitMQ implementation in Infrastructure.

## Why

Every workflow stage after this rides these contracts. Getting envelope shape (event ID, correlation ID, timestamp, type, payload) right once prevents N future migrations.

## Tasks

- [ ] Event envelope: event ID, correlation ID, occurred-at, event type, schema-versionable payload
- [ ] First contracts: `WorkOrderCreated`, `WorkOrderScheduled` (more arrive with their epics)
- [ ] `IEventPublisher` in Application; RabbitMQ implementation in Infrastructure
- [ ] Exchange/queue topology declared in code at startup
- [ ] Publish on the relevant API/domain actions

## Acceptance Criteria

- [ ] Scheduling a work order publishes a typed event observable in the RabbitMQ management UI
- [ ] Events carry correlation IDs originating from the API request
- [ ] Serialization round-trips cleanly (contract test)

## Decisions (locked at story start)

- **Contract location:** event contracts, the `EventEnvelope<T>`, and `IEventPublisher` live in **Application** (`Messaging/`) — the shared home, since both Api and Workers reference Application. The RabbitMQ implementation, `RabbitMqConfiguration`, and connection management live in **Infrastructure** (`Messaging/`). `RabbitMqConfiguration` moves from Api → Infrastructure so both hosts bind it.
- **Client library:** the official low-level **`RabbitMQ.Client`**, not MassTransit/EasyNetQ. The portfolio's point is to *show* the messaging plumbing; a framework would hide it.
- **Envelope shape:** `EventEnvelope<T>` record — `EventId` (Guid), `EventType` (stable string, e.g. `work-order.scheduled`), `SchemaVersion` (int), `CorrelationId` (Guid), `OccurredUtc` (DateTime), `Payload` (T). Serialized with System.Text.Json; self-describing over compact.
- **First contracts:** `WorkOrderCreated` (on create) and `WorkOrderScheduled`. "Scheduled" is the **Intake → Scheduled advance** — there is no separate schedule action in the state machine, so the handler publishes when an advance lands the order in `Scheduled`.
- **Publish point:** the Application handler publishes **after** a successful repository save (publish-after-commit). The domain aggregate stays pure (no infra dependency). Transactional/outbox delivery guarantees are Epic 8's concern.
- **Topology:** one **direct** exchange `artifice.events`; routing key = event type. Declared idempotently in code at startup (`ExchangeDeclare`). Queue/binding declaration for consumers arrives in 4.2.
- **Correlation split:** 4.1 adds the `CorrelationId` envelope field and a minimal stamp at the API boundary (`X-Correlation-ID` header, else generated) so events carry a real id. Full propagation into log scopes on both services + worker-side flow is **4.3**.
- **Testing split:** 4.1 ships a serialization round-trip *contract* unit test. The real-broker publish→consume integration test (Testcontainers RabbitMQ) lands in **4.2**, where a consumer exists to observe it.

## Notes

Decide envelope shape deliberately — it's also what the dashboard's event feed will render. Self-describing beats compact here.
