## Worker consumption and dispatch

**Labels:** messaging, backend
**Milestone:** M3

## Summary

Turn the empty Workers project into a hosted service that consumes events and dispatches them to typed handlers.

## Why

The Workers service is the other half of the async conversation; the handler-dispatch pattern established here is what every workflow epic plugs into.

## Tasks

- [ ] Hosted consumer service connecting to RabbitMQ with graceful startup/shutdown
- [ ] Handler abstraction (one handler per event type) with DI registration
- [ ] Manual acks: ack on success, nack/requeue policy defined (sophistication deferred to Epic 8)
- [ ] A first end-to-end proof: a handler for `WorkOrderScheduled` that logs and touches the order (e.g., appends a state-history note)
- [ ] Workers service added to CI build and covered by an integration test

## Acceptance Criteria

- [ ] Worker consumes a published event and its handler executes
- [ ] Unhandled handler exceptions don't kill the consumer loop
- [ ] Adding a new event handler requires no changes to the consumption plumbing
