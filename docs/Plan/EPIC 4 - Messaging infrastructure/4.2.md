## Worker consumption and dispatch

**Labels:** messaging, backend
**Milestone:** M3

## Summary

Turn the empty Workers project into a hosted service that consumes events and dispatches them to typed handlers.

## Why

The Workers service is the other half of the async conversation; the handler-dispatch pattern established here is what every workflow epic plugs into.

## Tasks

- [x] Hosted consumer service connecting to RabbitMQ with graceful startup/shutdown
- [x] Handler abstraction (one handler per event type) with DI registration
- [x] Manual acks: ack on success, nack/requeue policy defined (sophistication deferred to Epic 8)
- [x] A first end-to-end proof: a handler for `WorkOrderScheduled` that logs and touches the order (e.g., appends a state-history note)
- [x] Workers service added to CI build and covered by an integration test

## Acceptance Criteria

- [x] Worker consumes a published event and its handler executes — proven by `WorkerConsumerTests` (real RabbitMQ + Postgres via Testcontainers).
- [x] Unhandled handler exceptions don't kill the consumer loop — `OnReceivedAsync` catches everything and nacks `requeue:false`; the loop keeps running. (Guaranteed structurally; a dedicated poison-message test is a reasonable follow-up.)
- [x] Adding a new event handler requires no changes to the consumption plumbing — one `AddEventHandler<TEvent,THandler>()` call registers the handler and its routing-key binding; the consumer/dispatcher never reference concrete types.

## Decisions (inherited from 4.1, confirm at story start)

- **Workers becomes a Generic Host** (`Microsoft.NET.Sdk.Worker` / `Microsoft.Extensions.Hosting`) and gains a `ProjectReference` to **Infrastructure** (currently only references Application) so a handler can touch the order via a repository. Add it to the CI build.
- **Nack policy (deliberately simple):** on handler exception, **nack with `requeue: false`** and log — no requeue, because requeuing a poison message with no DLQ yet is an infinite loop. DLQ + retry sophistication is Epic 8. A dropped message is acceptable for the first slice.
- **End-to-end proof** carries the real-broker integration test deferred from 4.1: publish `WorkOrderScheduled` → worker consumes → handler appends a state-history note (Testcontainers RabbitMQ + Postgres).
