## Correlation and topology documentation

**Labels:** messaging, observability, docs
**Milestone:** M3

## Summary

Correlation IDs flowing API → event → worker → logs, and the broker topology documented.

## Why

Correlation is the thread the observability epic, the timeline view, and every debugging session will pull on. It must exist from the first async slice, not be retrofitted.

## Tasks

- [ ] Correlation ID established at (or accepted by) the API boundary per request
- [ ] Propagated through the event envelope and into all worker log scopes
- [ ] Structured logging includes correlation ID on both API and worker sides
- [ ] Topology doc: exchanges, queues, bindings, and the reasoning (lives in docs/, linked from README)

## Acceptance Criteria

- [ ] Grepping logs for one correlation ID tells one work order's whole story across both services
- [ ] A newcomer can draw the broker topology from the doc without reading code

## Scope clarification (vs 4.1)

4.1 already establishes the `CorrelationId` envelope field and a minimal API-boundary stamp (`X-Correlation-ID` header, else generated). This story **hardens** that: pushes the id into structured-logging scopes on **both** the API and worker sides, propagates it from the consumed envelope into the worker's log scope, and writes the topology doc (exchanges, queues, bindings + reasoning) in `docs/`, linked from README.
