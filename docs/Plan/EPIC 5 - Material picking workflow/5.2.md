## Material picking consumer (happy path)

**Labels:** messaging, backend, domain

## Summary

Replace the `WorkOrderScheduledHandler` stand-in with the first real workflow behaviour: on `WorkOrderScheduled`, resolve the product's BOM, reserve the component materials it calls for, record the pick, and emit an event that lets the pipeline continue to production. This story is the happy path only — sufficient stock, single delivery, no contention. The hard guarantees land in 5.3 and 5.4.

## Why

This is the milestone the whole M3 arc has been building toward: a work order flows API → RabbitMQ → worker → *a real factory action* → back onto the bus. After this story the demo is genuinely event-driven — scheduling an order visibly consumes inventory — and Epic 6 has an event to hang production off.

## Tasks

- [ ] Domain reservation behaviour: given a BOM and requested quantity, compute component demand and reserve it against on-hand inventory (happy-path decrement; the all-or-nothing/insufficient guard is 5.3)
- [ ] A pick record — `MaterialReservation` (per work order, with the reserved lines) — persisted so the pick is auditable and 5.4 has a natural idempotency key
- [ ] Rewrite `WorkOrderScheduledHandler` to load the order + product BOM, reserve, persist the reservation, and append a state-history note describing what was picked
- [ ] New event contract `MaterialsReserved` (work order id, reserved lines) published after the reservation commits
- [ ] Repository/query support the handler needs (load product with BOM; load/insert reservation)
- [ ] Integration test: publish `WorkOrderScheduled` → worker reserves per BOM → on-hand drops, reservation row exists, history note written, `MaterialsReserved` observed (Testcontainers RabbitMQ + Postgres)

## Acceptance Criteria

- [ ] A scheduled work order triggers material reservation via the consumed event, not an API call
- [ ] On-hand inventory for each BOM component drops by `QtyPerUnit × OrderItemQty`
- [ ] The pick is visible in the work order's state history and in a persisted reservation record
- [ ] A `MaterialsReserved` event is published so Epic 6's production stage has something to consume
- [ ] The old "scheduling acknowledged" stand-in behaviour is fully replaced

## Decisions (to confirm at story start)

- **Picking does not itself advance the work order Scheduled → InProcess.** That advance is production starting, which Epic 6 owns. On a successful pick the order stays `Scheduled` with a reservation recorded and history noted; the emitted `MaterialsReserved` is what Epic 6's production consumer will act on. This keeps the epic boundary clean and the pipeline flowing without 5.x reaching into 6.x's state transition.
- **Reservation is modelled as its own record (`MaterialReservation`)**, keyed by work order, rather than mutating the work-order aggregate's stock list. The existing `AssignedStock`/`StockKeepingUnit` on `WorkOrder` represents *finished serialized units* (Epic 6 territory), which is a different concept from *component materials consumed to build them* — don't conflate the two.
- **Where the reservation transaction lives:** an Application-layer handler/service the worker calls, so the domain rule (compute demand, decrement) stays testable without infrastructure. Confirm against the existing handler-dispatch shape from 4.2.
- **`MaterialsReserved` follows the established envelope conventions** (self-describing, serialization-friendly) — the dashboard event feed (Epic 11) will render it.

## Notes

Keep the write path deliberately naive here — a straight read-modify-write decrement is fine for the happy path. Do **not** pre-solve concurrency or duplicate delivery in this story; 5.3 and 5.4 exist precisely so those guarantees get their own focused design and tests. Resist the urge to gold-plate the reservation transaction before its failure modes have stories.
