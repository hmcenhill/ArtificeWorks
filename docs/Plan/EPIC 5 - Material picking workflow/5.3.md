## Insufficient stock and concurrency-safe reservation

**Labels:** messaging, backend, domain, reliability

## Summary

Harden the reservation the happy path (5.2) established into a correct one: reservation is all-or-nothing, insufficient stock puts the work order OnHold with a visible reason (releasing nothing it didn't reserve), and two work orders racing for the same scarce stock can never both win. This is the story where the classic event-driven inventory problem — reservation under concurrency — is confronted head-on.

## Why

This is the point of the epic. A factory that reserves materials but happily oversells stock or leaks half-reservations isn't a factory. Getting all-or-nothing reservation and no-double-allocation right, and *proving* it under real contention, is exactly the reliability-engineering story the portfolio is meant to show.

## Tasks

- [ ] All-or-nothing reservation: if any BOM line can't be fully satisfied, reserve nothing and leave every component's on-hand untouched
- [ ] Insufficient-stock outcome: place the work order `OnHold` via `SetHold` with a reason naming the short component(s); the reason lands in state history
- [ ] Concurrency guard on the on-hand decrement so concurrent picks cannot both draw the same stock below zero (see Decisions for the mechanism)
- [ ] Decide the message-side outcome on insufficient stock: the message is *handled* (order is OnHold), so it acks — it is not an error/nack. Document why (it's a business outcome, not a transient fault)
- [ ] Integration tests (Testcontainers Postgres + RabbitMQ):
  - [ ] insufficient stock → order OnHold with reason, on-hand for every line unchanged
  - [ ] N concurrent work orders against stock enough for fewer than N → exactly the affordable number reserve, the rest go OnHold, and total reserved never exceeds on-hand (no double-allocation)

## Acceptance Criteria

- [ ] Insufficient stock is handled cleanly and visibly: the work order is OnHold and the shortage reason appears in its state history
- [ ] A failed reservation releases nothing it didn't reserve — partially-available BOMs leave on-hand exactly as it was
- [ ] Concurrent work orders cannot reserve the same stock twice; under contention, total reserved for any component never exceeds its on-hand
- [ ] The insufficient-stock path acks the message (business outcome), it does not nack/drop it

## Decisions (to confirm at story start)

- **Concurrency mechanism — recommended: an atomic conditional decrement** (`UPDATE components SET on_hand = on_hand - @qty WHERE id = @id AND on_hand >= @qty`, checking rows-affected). It pushes the guard into the database, needs no aggregate rowversion, reads clearly in the portfolio write-up, and composes with all-or-nothing (any line's conditional update affecting 0 rows aborts the whole reservation and rolls back). The alternative — optimistic concurrency (`xmin`/rowversion) with retry — is heavier and the codebase has *no* optimistic concurrency today (see the standing HANDOFF open decision on `WorkOrder`); introducing it here would be a bigger commitment than the slice needs.
- **All-or-nothing is transactional:** reserve all lines inside one DB transaction; if any conditional decrement fails the short check, roll the whole transaction back so no component is left decremented. The reservation record and every decrement commit together or not at all.
- **Insufficient stock is a business outcome, not a failure:** the handler succeeds (order → OnHold, reason recorded) and the message is acked. Reserve nacks for genuine transient faults so 5.x doesn't muddy the Epic 8 retry/DLQ story.
- **Ordering/deadlock:** if a single reservation decrements multiple components, acquire them in a deterministic order (e.g. by component id) to avoid deadlocks between concurrent multi-line reservations. Confirm whether this is needed given the conditional-update approach.

## Notes

The concurrency test is the centerpiece — write it to actually race (fire N reservations in parallel against deliberately scarce stock) and assert the invariant `sum(reserved) <= on_hand` holds, not just that it usually does. This test is also the seed of Epic 12's "kill a worker mid-pick" failure-injection demo, so make the reservation boundary explicit and observable.
