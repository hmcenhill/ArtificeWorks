## Idempotent consumption

**Labels:** messaging, backend, reliability

## Summary

Make picking safe against duplicate deliveries: the same `WorkOrderScheduled` delivered twice reserves materials once. This closes the last acceptance criterion of the epic and the last of the "classic event-driven problems" it set out to confront.

## Why

The messaging backbone is deliberately at-most-once *publish* but at-least-once *delivery* is the world any real broker lives in — redeliveries happen on nack-requeue, consumer restarts, and network hiccups. Without idempotent consumption, a redelivered scheduling event would double-decrement inventory: the exact bug event-driven systems are infamous for. Proving this one is handled is a headline portfolio point.

## Tasks

- [x] Idempotency mechanism keyed on the event/work-order so a second delivery is a no-op (see Decisions for the two candidate approaches)
- [x] The picking handler consults it before reserving and records having processed the event as part of the same transaction as the reservation
- [x] Confirm the duplicate path still acks (it's handled, not failed)
- [x] Integration test: deliver the same `WorkOrderScheduled` twice → inventory decrements once, exactly one reservation record, no duplicate history note (Testcontainers)
- [x] Note in the state history / logs when a duplicate is skipped, so the dashboard event feed can show idempotency working live (Epic 11/12)

## Acceptance Criteria

- [x] Duplicate deliveries of the same event have no additional effect: on-hand decrements once, one reservation exists, no second pick note
- [x] The dedupe decision commits atomically with the reservation (no window where the reservation commits but the dedupe marker doesn't, or vice-versa)
- [x] A skipped duplicate is observable (logged / noted), not silently swallowed

## Decisions (to confirm at story start)

- **Two candidate mechanisms — recommended: the reservation record *is* the idempotency key.** Put a unique constraint on `MaterialReservation.WorkOrderId` (one pick per work order); a duplicate delivery's insert hits the constraint, the handler treats the conflict as "already picked," and acks. No extra table, and the dedupe marker and the reservation are literally the same row — atomicity is free. Works because picking is naturally once-per-work-order.
  - *Alternative:* a generic processed-events / inbox table keyed on `EventId`, written in the reservation transaction. More general (handles any event type, groundwork for a shared idempotency helper), but heavier and broader than this epic needs. Prefer the reservation-as-key approach now; revisit a shared inbox if Epic 6+ handlers need the same guard and it's worth generalizing.
- **Keyed on work order, not raw `EventId`:** the guarantee we want is "this order is picked once," which is stronger and more meaningful than "this exact message id processed once" (a legitimately re-published-with-new-id scheduling event should still not double-pick). Confirm this matches the intended semantics.
- **Duplicate = success, ack it.** Same stance as 5.3: a duplicate is a handled outcome, not a fault to nack.

## Notes

This is small but conceptually load-bearing — it's the "idempotent consumption" line item from the epic summary and one of the reliability behaviours Epic 12's failure injection will let a visitor trigger on purpose ("redeliver this message" → watch nothing double-count). Keep the skip observable for exactly that reason. With this story the epic's five acceptance criteria are all met and the first async workflow stage is complete end-to-end.
