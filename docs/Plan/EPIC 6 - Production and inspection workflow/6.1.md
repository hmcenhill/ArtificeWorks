## Serialized units and the production consumer

**Labels:** messaging, backend, domain

## Summary

Bind the first consumer to `work-order.materials-reserved` — the routing key Epic 5 has been publishing to nobody — and make production real: advance the order Scheduled → InProcess, build the ordered quantity as serialized `StockKeepingUnit`s owned by the work order, and publish `work-order.production-completed`. Happy path only: everything builds, nothing fails, single delivery.

## Why

This closes the loop Epic 5 deliberately left open. Picking hands over by publishing rather than transitioning, so until something binds that key the pipeline stops at Scheduled. This story is also where `StockKeepingUnit` stops being a decorative record and becomes a stateful thing with a lifecycle — everything in 6.2 and 6.3 hangs off that.

## Tasks

- [ ] `StockKeepingUnit` gains a lifecycle: a `UnitStatus` (`Built` / `Passed` / `Scrapped`), `BuiltUtc`, `InspectedUtc?`, `ScrapReason?`, and the build attempt that produced it. Only `Built` is reachable in this story
- [ ] Change the work-order ↔ SKU relationship from many-to-many (`work_order_skus`) to a **one-to-many** owned collection with a `work_order_id` FK on `skus` — a serialized unit belongs to exactly one order, and 6.3 needs to count this order's passing units cheaply
- [ ] `WorkOrder` production behaviour: build the outstanding quantity, appending units to `AssignedStock` and advancing Scheduled → InProcess. Fix `AssignSku`'s fulfilment guard, which currently counts *all* assigned units — from 6.3 it must count only non-scrapped ones
- [ ] `Application/Production/ProductionService` owning the workflow (load order → dedupe pre-check → build units → advance status → note history → publish), mirroring `MaterialPickingService`'s shape
- [ ] Worker handler `MaterialsReservedHandler` as a thin adapter: adopt the inbound correlation id, call the service, log the outcome — same shape as `WorkOrderScheduledHandler`
- [ ] New event contract `ProductionCompleted` (`work-order.production-completed`) carrying the work order id, the serial numbers built, and the attempt number
- [ ] Migration for the SKU columns and the FK change. **Fold in the long-outstanding `WorkOrderStateHistory.CompletedBy` mapping** while a migration is already being written (see HANDOFF open decisions) — it has been silently unmapped since 4.2
- [ ] Integration test: publish `WorkOrderScheduled` → picking reserves → production builds → order is InProcess, `OrderItemQty` SKU rows exist against it, history notes the build, `ProductionCompleted` observed on the bus with the original correlation id

## Acceptance Criteria

- [ ] A work order with materials reserved advances Scheduled → InProcess via the consumed event, not an API call
- [ ] Exactly `OrderItemQty` serialized units exist against the order, each `Built`, each with a distinct serial number
- [ ] The build is visible in the work order's state history
- [ ] `ProductionCompleted` is published so 6.2's inspection stage has something to consume
- [ ] The full chain API → scheduled → picked → produced runs under one correlation id

## Decisions (to confirm at story start)

- **Production completes instantly.** No delay, no timer, no `due at` column. Epic 10's simulation engine owns pacing; a sleeping handler here would block the prefetch-1 consumer and buy nothing. Settled at grooming.
- **Units are built, not allocated from stock.** `AssignSku`/`UnassignSku` exist and imply picking finished units off a shelf. This factory builds to order — production *creates* the serialized unit. Keep `AssignSku` as the internal mechanism if convenient, but the domain verb is "build".
- **`WorkOrder.Cancel` clears `_assignedStock`** to "return" units. With an owning FK that becomes an orphan/delete question rather than a detach. Decide it here: most likely the units stay, marked scrapped or simply retained as history of what was built — a cancelled order silently deleting serialized units is worse than keeping them.
- **The state-history note is written outside the build transaction**, consistent with the deliberate choice made in 5.2. Audit stays out of the hot path.

## Notes

Keep this story naive in the same way 5.2 was: single delivery, no contention, no failure. The consumer will legitimately run *again* for the same order in 6.3 (a rebuild), which is exactly what makes 6.4's idempotency story interesting — do not pre-solve it here beyond a cheap pre-check.
