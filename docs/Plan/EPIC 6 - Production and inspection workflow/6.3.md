## Scrap, rebuild, and the Fault cap

**Labels:** messaging, backend, domain

## Summary

Close the loop. When inspection leaves the order short of its ordered quantity, publish `work-order.rework-required`; the production consumer picks it up, rebuilds **only the shortfall**, and the order goes round again. A configurable cap on rebuild attempts stops the cycle: exceeding it sends the order to Fault with the scrap history as its reason.

## Why

This is the epic's headline: a real cycle in the state machine, driven entirely over the bus, with a bounded escape hatch. It is also the first *legitimate* use of Fault — the state has existed since Epic 2 waiting for a failure that isn't a bug — and the first backward transition the work order has ever made.

## Tasks

- [ ] `WorkOrder` backward transition `ReturnToProduction(by, notes)`: Inspection → InProcess. `AdvanceToNextStep` only walks forward via `GetNextStatus`, so this is new and needs its own guards (only from Inspection, never from a terminal state)
- [ ] `WorkOrder` derived quantities: passing-unit count and outstanding quantity (`OrderItemQty` − passed), plus the current build attempt. These are what both the rebuild and the "are we done?" check read
- [ ] Rebuild path in `ProductionService`: build the **outstanding** quantity, not the full quantity, incrementing the attempt number
- [ ] New event contract `ReworkRequired` (`work-order.rework-required`) carrying the work order id, the scrapped serial numbers with reasons, the outstanding quantity, and the attempt number. Bind it on the production consumer alongside `materials-reserved` — both funnel into the same service entry point
- [ ] Rebuild cap: `Production:MaxRebuildAttempts`, default 3. On exceeding it, `SetStatus` the order to Fault with a summary naming the scrapped units, publish `WorkOrderFaulted` (`work-order.faulted`), and **stop** — no further rework event
- [ ] Confirm the recovery path works: a Faulted order is already cancellable (Fault is deliberately non-terminal) and releasable. Verify releasing it does something sensible, or document what it does not do, in the same spirit as the 5.3 release-does-not-re-pick note
- [ ] Tests: unit tests for the backward transition guards, outstanding-quantity arithmetic, and the cap boundary (attempt N passes, N+1 faults); integration test driving a forced failure through a full rebuild cycle to Delivery, and a second forcing repeated failures to Fault

## Acceptance Criteria

- [ ] A failed inspection scraps the unit with its reason and returns the order to InProcess
- [ ] A rebuild builds only the shortfall — passing units from earlier attempts are never rebuilt or re-inspected
- [ ] The order reaches Delivery only once the full ordered quantity has passed
- [ ] Exceeding the rebuild cap places the order in Fault with a reason naming what was scrapped, and the cycle stops
- [ ] A Faulted order remains recoverable (cancellable, and releasable to a documented outcome)
- [ ] Every scrap, rebuild, and transition is in the state history — the trail explains the whole cycle without the logs

## Decisions (to confirm at story start)

- **Per-unit scrap and rebuild of the shortfall only**, settled at grooming. Passing units are kept.
- **The cap counts rebuild *attempts*, not scrapped units.** An order that scraps three units in one attempt has used one attempt. Attempts are the loop bound; units are the symptom.
- **⚠️ Rebuilds consume no new materials — confirm this before writing code.** Physically a scrapped unit burns its parts, so a rebuild should re-pick. But Epic 5 made `material_reservations.WorkOrderId` **unique** as its idempotency key (5.4), so a second reservation for the same order is impossible without reopening that design and adding a migration. Three ways out, in order of preference:
  1. **Rebuilds consume nothing** — the original pick covers the order, scrapped parts are notionally salvaged. Zero schema change; document it as a known simplification for Epic 13.
  2. **Widen the key** to `(WorkOrderId, AttemptNumber)` so each attempt gets its own reservation. Honest, and it composes with 6.4's dedupe key, but it costs a migration and a revisit of 5.4's argument.
  3. **Defer entirely** — rebuild on the original pick now, add material consumption in Epic 13 with multi-level BOMs.

  This is the one open question the grooming did not close. Raise it at story start.
- **Fault is set via `SetStatus`, the superuser command.** There is no domain verb for "this went wrong" yet. If `SetStatus` feels wrong to use from a worker, that is a signal to add a proper `Fault(reason)` transition rather than to work around it.
