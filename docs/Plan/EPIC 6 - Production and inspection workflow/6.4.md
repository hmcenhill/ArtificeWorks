## Idempotent consumption for repeating stages

**Labels:** messaging, backend, reliability

## Summary

Make the production and inspection consumers safe against redelivery. Epic 5 solved this elegantly and cheaply — the reservation row *was* the dedupe key, because picking happens exactly once per order. **That trick does not transfer**: production legitimately runs many times for one work order, so "has this order been produced?" is no longer a question with an answer. This story finds the key that does work.

## Why

At-least-once delivery is a fact of the transport, not an Epic 8 concern — a consumer restart mid-handler redelivers today. A duplicate `ProductionCompleted` that builds a second batch of units, or a duplicate `ReworkRequired` that burns a rebuild attempt, corrupts the order quietly. And unlike 5.4, there is no free unique index to lean on, which makes this the more interesting version of the problem.

## Tasks

- [x] Establish the dedupe key. The natural candidate is **`(WorkOrderId, AttemptNumber)`** — an attempt is exactly the thing that should happen once — with the source `EventId` as the alternative if attempts turn out to be inferred rather than carried on the wire. Whichever wins, prefer a **unique constraint that commits with the work** over a separate marker table, keeping 5.4's "the record is the key" property
- [x] Production: a redelivered `MaterialsReserved` or `ReworkRequired` for an attempt already built must build no units, advance nothing, and write no history note (per 5.4, a note per redelivery is itself non-idempotent)
- [x] Inspection: a redelivered `ProductionCompleted` must not re-verdict units already inspected. The `already inspected` guard from 6.2 does most of this — verify it holds when the *whole batch* is redelivered, and that no second `InspectionPassed` or `ReworkRequired` goes out
- [x] Confirm the ack/nack line stays sharp: duplicates ack, scrap and Fault ack (business outcomes), only genuine faults nack — extend the table in [docs/messaging-topology.md](../../messaging-topology.md) with the production and inspection rows
- [x] Log every skipped duplicate at information level, never silently — Epic 12 will redeliver a message on purpose in front of an audience and needs it observable
- [x] Tests, following the 5.4 shape: sequential redelivery and simultaneous redelivery for both consumers, plus a redelivery landing *mid-rebuild-cycle* (the case Epic 5 never had) proving the attempt counter cannot be double-advanced
- [x] Update `docs/messaging-topology.md`: new queues/bindings, the full event set, the rework cycle, and a short section on why the Epic 5 dedupe key did not generalise

## Acceptance Criteria

- [x] Redelivery of any production or inspection event has no additional effect on units, status, attempt count, or history
- [x] Duplicates are acked and logged, never nacked and never silent
- [x] The guarantee rests on a database constraint, not on a check-then-act read
- [x] Concurrent duplicate deliveries are proven safe, not just sequential ones
- [x] `docs/messaging-topology.md` describes the Epic 6 topology accurately enough to redraw from

## Decisions (to confirm at story start)

- **The dedupe key is attempt-scoped, not order-scoped.** This is the whole point of the story. An order-scoped key would make the rebuild loop from 6.3 impossible.
- **Prove the guarantees where they can actually be proven.** 5.3 learned that the broker path *cannot* demonstrate concurrency, because prefetch 1 on a single consumer serializes deliveries — so race the service directly in parallel scopes against Postgres, and let the broker-backed test own the plumbing claim end-to-end. Reuse that split rather than rediscovering it.
- **No inbox table unless the key genuinely needs one.** A separate marker means a second write that can diverge from the work; the constraint-on-the-work-itself approach from 5.4 is better and should be exhausted first.

## Notes

If task one lands on `(WorkOrderId, AttemptNumber)`, it interacts with 6.3's open materials question: widening `material_reservations` to the same key would make reservations, builds, and dedupe all attempt-scoped and consistent. Worth noticing before choosing, not after.

This story closes Epic 6. After it, the pipeline runs Intake → Scheduled → InProcess → Inspection → Delivery unattended, with a bounded, visible, recoverable failure loop in the middle — and Epic 7 has `work-order.inspection-passed` waiting for a subscriber, exactly as Epic 5 left `materials-reserved` for this one.
