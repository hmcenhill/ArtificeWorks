## Shipment booking and the carrier

**Labels:** messaging, backend, domain

## Summary

Bind a consumer to `work-order.inspection-passed` — the routing key Epic 6 has been publishing to nobody — and book a virtual carrier for the units that passed: create a `Shipment` with a carrier, a tracking number and an estimated arrival, and publish `work-order.shipment-scheduled`. Happy path only: the carrier always accepts, single delivery, no dispatch yet.

## Why

This closes the loop Epic 6 deliberately left open, in exactly the way 6.1 closed Epic 5's. It is also where the shipping stage's own progress gets somewhere to live: the work order is **already in Delivery** when this consumer runs (`InspectionService.Resolve` advanced it), so booking transitions nothing on the aggregate. That is a feature, not a gap — the work order tracks manufacturing, the `Shipment` tracks the parcel, and 7.2's dispatch is what finally moves the order to Completed.

## Tasks

- [x] Domain `Shipment` aggregate: `Id`, `WorkOrderId`, `Carrier`, `TrackingNumber`, `BookedUtc`, `EstimatedArrivalUtc`, `DispatchedUtc?`, and a `ShipmentStatus` (`Booked` / `Dispatched`). Only `Booked` is reachable in this story. Modelled as its own aggregate next to `MaterialReservation`, for the same reason: it is about the parcel, not about the order's manufacturing state
- [x] `ShipmentLine` per serialized unit shipped (serial number), mirroring `MaterialReservationLine` — the record of *which* units went out, which matters once an order has scrapped some and rebuilt others
- [x] `ICarrierBooking` + `CarrierBookingResult` (accepted with carrier / tracking / ETA, or refused with a reason) and a `ConfiguredCarrierBooking` implementation — deliberately the same swappable shape as 6.2's `IVerdictSource`, so 7.3's refusals and Epic 12's injection have a seam that already exists. This story only ever takes the accepting branch
- [x] `ShippingConfiguration`: `Shipping:Carriers` (a list of virtual carriers, seeded with a couple of in-world names), `Shipping:TransitDays` (default 3, feeds the ETA), `Shipping:Seed` for deterministic tests. `RefusalRate` arrives in 7.3
- [x] `Application/Shipping/ShippingService.BookShipment` owning the workflow — load order → dedupe pre-check → verify the order is in Delivery and fulfilled → book → persist → note history → publish — mirroring `MaterialPickingService` and `ProductionService`
- [x] Worker handler `InspectionPassedHandler` as a thin adapter: adopt the inbound correlation id, call the service, log the outcome
- [x] New event contract `ShipmentScheduled` (`work-order.shipment-scheduled`) carrying work order id, carrier, tracking number, ETA and the shipped serial numbers
- [x] **Idempotency, and it is cheap again**: a unique index on `shipments.work_order_id`. Shipping happens exactly once per order, so Epic 5's "the record *is* the dedupe key" applies unchanged — the row and its lines commit together and a duplicate delivery's insert collides. Cheap pre-check for the common case, constraint for the race, duplicates ack and log
- [x] Migration `Shipping` for the two tables and the unique index
- [x] Integration test over the broker: create → advance → picked → built → inspected → **shipment booked**, under one correlation id, with `ShipmentScheduled` observed on the bus

## Acceptance Criteria

- [x] An order whose full quantity passed inspection gets a shipment booked by the consumed event, not an API call
- [x] The shipment names a carrier, a tracking number and an estimated arrival, and lists exactly the passed units' serial numbers
- [x] The booking is visible in the work order's state history
- [x] `ShipmentScheduled` is published so 7.2's dispatch consumer has something to consume
- [x] Redelivery of `InspectionPassed` books no second shipment, is acked, and is logged
- [x] The whole chain API → scheduled → picked → built → inspected → booked runs under one correlation id

## Decisions (to confirm at story start)

- **Booking does not transition the work order.** It is already in Delivery. Resist adding a status for "shipment booked" — the parcel's lifecycle belongs on the `Shipment`, and adding a manufacturing status for a logistics fact would put two clocks on one aggregate.
- **The shipment covers the passed units only.** Scrapped serials never ship. `InspectionPassed` already carries the passing serial numbers; prefer trusting the event's payload over re-deriving from the order, so a redelivery would describe the same parcel.
- **Two steps, not three.** Book here, dispatch in 7.2, and stop. An in-transit stage with a delivery confirmation would need a timer or sweeper, and Epic 6 deferred all pacing to Epic 10's simulation engine — settled at grooming.
- **Booking is instant**, for the same reason production was.

## Notes

Keep this naive the way 5.2 and 6.1 were: acceptance only, no refusal, no manual path. 7.3 turns the carrier hostile, and the `ICarrierBooking` seam is the only concession this story makes to that.

Worth noticing while writing it: the dedupe key going back to order-scoped is not a regression to Epic 5, it is the honest answer to a stage that genuinely happens once. 6.4's whole argument was that the key should follow *the thing that must happen once* — here that is the order again.
