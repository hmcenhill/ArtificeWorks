## Dispatch, completion, and the visitor's carrier choice

**Labels:** messaging, backend, api

## Summary

The pipeline's ending. A second consumer binds `work-order.shipment-scheduled`, dispatches the parcel, advances Delivery → **Completed**, and publishes `work-order.completed` — the last event in the chain. Alongside it, the human path: turn `Shipping:AutoBook` off and an order sits in Delivery waiting for a visitor to choose a carrier via `POST /work-orders/{id}/shipments`, which books the shipment and rejoins the identical flow.

## Why

Completion is the payoff — the demo's story needs an ending, and until something advances Delivery → Completed the pipeline stops one step short. The manual booking path exists for the same reason 6.2's verdict endpoint does: an unattended pipeline is a nice claim but a visitor who can only watch has no reason to stay. Carrier choice is the natural decision moment in a stage that otherwise has none.

## Tasks

- [ ] `ShippingService.DispatchShipment`: load the shipment, mark it `Dispatched` with `DispatchedUtc`, advance the work order Delivery → Completed via `AdvanceToNextStep`, note the history, publish `WorkOrderCompleted`
- [ ] Worker handler `ShipmentScheduledHandler`, same thin-adapter shape as the rest
- [ ] New event contract `WorkOrderCompleted` (`work-order.completed`) carrying work order id, product id, carrier, tracking number and shipped serial numbers. **One event, not two** — `shipment-dispatched` and `completed` would say the same thing at the same instant. Nothing binds it; it is the terminal announcement, for Epic 11's dashboard
- [ ] `Shipping:AutoBook` (default **true**), mirroring `Inspection:AutoInspect`. Off, the `InspectionPassed` handler records that the order awaits a carrier and stops; the order rests in Delivery with no shipment
- [ ] `POST /work-orders/{id}/shipments` — optional `carrier` in the body (omitted ⇒ the configured booking source picks), `createdBy` as elsewhere. It calls **the same `BookShipment`** 7.1's consumer calls, so an order shipped by a visitor and one shipped unattended reach identical state, exactly as 6.2's two paths do
- [ ] Stable ProblemDetails codes for the endpoint: `order_not_in_delivery` (409), `shipment_already_booked` (409), `unknown_carrier` (400). Extend `ProblemCodes` in the existing style
- [ ] Surface the shipment on the work-order read model (`WorkOrderDto`): carrier, tracking, status, booked/dispatched/ETA timestamps, shipped serials
- [ ] Dispatch idempotency: a redelivered `ShipmentScheduled` must dispatch nothing twice. The `Shipment`'s own status transition is the guard (`Booked` → `Dispatched` once, refused otherwise) — no new table. Verify the belt-and-braces case too: `Completed` is terminal, so the order's own `AdvanceToNextStep` would reject a second advance anyway. Duplicates ack and log
- [ ] Update the ack/nack table in [docs/messaging-topology.md](../../messaging-topology.md) with the shipping and dispatch rows
- [ ] Integration tests: (a) unattended — one `advance` call and the order reaches **Completed** over the bus with `WorkOrderCompleted` observed under the original correlation id; (b) `AutoBook: false` — the order waits in Delivery, the endpoint books it, and it still completes; (c) redelivered `ShipmentScheduled` changes nothing

## Acceptance Criteria

- [ ] A dispatched shipment completes the work order, driven by an event
- [ ] `work-order.completed` is published once per order, carrying the shipment's details
- [ ] With `AutoBook` off, an order waits in Delivery until a carrier is chosen, then completes normally
- [ ] The manual and automatic booking paths produce indistinguishable end state
- [ ] The endpoint returns stable problem codes for a wrong-state order, an already-booked order, and an unknown carrier
- [ ] Redelivery of `ShipmentScheduled` dispatches nothing twice
- [ ] The shipment is visible on `GET /work-orders/{id}`

## Decisions (to confirm at story start)

- **Completion is automatic once a shipment is dispatched.** The visitor's decision is *which carrier*, not *whether to finish*. A second confirmation step would be a click with nothing behind it.
- **Cancelling an order with a booked, undispatched shipment.** `Cancel` currently knows nothing about shipments; a cancelled order should not leave a live parcel. Cheapest honest answer: void the shipment (a `Cancelled` status on it) when the order is cancelled. Decide it here rather than discovering it in Epic 12 — and note that after dispatch the question disappears, because `Completed` is terminal and refuses `Cancel` already.
- **No `Shipping:AutoBook` equivalent for dispatch.** Booking is the decision; dispatch is bookkeeping. Two switches for one stage would be noise.

## Notes

After this story the demo has a complete sentence: a visitor creates an order, presses advance once, and watches it go Intake → Scheduled → InProcess → Inspection → Delivery → Completed with a parcel and a tracking number at the end — or takes the wheel at inspection and at the carrier and drives it by hand. That is the M4 milestone's headline.
