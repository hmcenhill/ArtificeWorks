## Carrier refusal, hold, and making release mean something

**Labels:** messaging, backend, reliability

## Summary

Give the carrier the right to say no: a refused booking puts the work order OnHold with the reason, rather than faulting or silently stalling. Then close the loop that has been open since 5.3 — **releasing an order held at Delivery re-requests the booking**, so a hold becomes a recoverable pause instead of a dead end a human has to nurse.

## Why

The shipping stage needs a failure path, and refusal is the honest one: no carrier capacity is an external constraint, not a defect, so OnHold (not Fault) is the right state — the same reading 5.3 took for a material shortage.

The second half is the more valuable half. "Releasing a held work order does not re-trigger anything" has been a standing open decision through two epics, and every hold in the system is currently a state change that leads nowhere. Delivery is the right place to fix it: the trigger is a single event with no attempt counter and no partial work behind it, which makes it the least dangerous re-trigger in the pipeline to build first.

## Tasks

- [x] `Shipping:RefusalRate` (default **0.0**, so the unattended pipeline still flows) and `Shipping:RefusalReason`, wired through the `ICarrierBooking` seam 7.1 established. `Shipping:Seed` makes it deterministic for tests
- [x] Refusal path in `ShippingService.BookShipment`: no shipment row, `SetHold` naming the carrier and the reason, the reason in state history, and the message **acked** — a factory waiting on a carrier is a result, not an error, exactly as an insufficient-stock hold is
- [x] The manual endpoint from 7.2 reports refusal as its own outcome — `carrier_unavailable` (409), distinct from `unknown_carrier` (400): the carrier exists, it just won't take the job
- [x] **Release re-triggers shipping.** When `ReleaseWorkOrder` restores an order to Delivery *and it has no shipment*, publish `InspectionPassed` again (carrying the currently passing serials). The existing consumer picks it up and books — no new routing key, no new consumer, and the idempotency from 7.1 covers the case where a shipment appeared in the meantime
- [x] Keep the re-trigger narrow and explicit: **only** a release that lands in Delivery with no shipment republishes. Releases into any other status stay inert, as 5.3 and 6.3 left them — this story does not quietly re-arm the whole pipeline
- [x] Record the re-publish in state history and log it, so the recovery is visible rather than magic
- [x] Tests: refusal → OnHold with the named reason and an ack; release → re-book → dispatch → Completed, end to end over the broker; release of an order that already has a shipment publishes nothing; a refusal followed by a redelivery does not double-hold
- [x] Update [docs/messaging-topology.md](../../messaging-topology.md): the refusal row in the ack/nack table, and a short note that `work-order.inspection-passed` now has a **second publisher** (the API, on release) as well as the inspection worker

## Acceptance Criteria

- [x] A refused booking holds the order with the carrier's reason and acks the message
- [x] No shipment row exists for a refused booking
- [x] Releasing an order held at Delivery without a shipment re-requests booking and, with the carrier willing, completes the order
- [x] Releasing in any other circumstance publishes nothing — proven, not assumed
- [x] The re-request is visible in state history and the log
- [x] `Shipping:RefusalRate` at 0.0 leaves the unattended pipeline unchanged

## Decisions (to confirm at story start)

- **Refusals leave no row.** Persisting refused bookings would break 7.1's unique index on `shipments.work_order_id` and turn a clean retry into a "find the live one" query. The cost: the count of refusals lives only in state history. Accepted; if refusal analytics ever matter, that is a separate `booking_attempts` table, not a status column on `shipments`.
- **Refusal is uncapped.** Unlike 6.3's rebuild cap, a carrier saying no costs nothing and consumes no materials, and each retry is a deliberate human act (a release), not an automatic loop. So there is no runaway to bound — but re-check that reasoning if Epic 10's simulation engine ever releases holds on its own, because that turns a human act into a loop.
- **The API becomes an event publisher for a stage it doesn't own.** Slightly uncomfortable, and worth stating plainly: the alternative is a `shipping-requested` key with the same consumer behind it, which is more honest but is one more contract for one more caller. Prefer re-publishing `InspectionPassed` unless writing it feels wrong, and if it does, prefer the new key over a clever workaround.

## Notes

This story quietly answers a question the HANDOFF has carried since Epic 5 — whether release should re-trigger — and answers it *narrowly*. The general answer (releases anywhere re-arm the pipeline) belongs to Epic 10, where the simulation engine can own the retry policy. What this proves is that the mechanism works and the idempotency guarantees hold up when a stage is re-entered on purpose.
