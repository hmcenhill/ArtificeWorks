## The work order timeline

**Labels:** api, backend, demo

## Summary

One endpoint that tells a work order's whole story in order: `GET /work-orders/{id}/timeline` returns a single chronological list of typed entries — state changes, the material pick, each build attempt, each inspection and its verdicts, the shipment booking and dispatch — instead of making a caller stitch four endpoints together and sort by hand.

## Why

The system now has a genuinely interesting story to tell per order (picked these components, built three units, scrapped one, rebuilt it, shipped with this tracking number) and no single place that tells it. This is also the seed of Epic 11's trace view: getting the *shape* right now means the dashboard renders a list rather than reconciling five read models in TypeScript.

## Tasks

- [ ] Decide the entry shape and commit to it. Proposal: a flat array ordered by timestamp, each entry `{ at, kind, by?, summary, detail }` with `kind` in `state` / `pick` / `build` / `inspection` / `verdict` / `shipment`, and `detail` a per-kind object. A discriminated list renders as one column with typed icons; parallel arrays would not
- [ ] Read model + query: compose from what is already persisted — `work_order_state_history`, the `material_reservation` and its lines, `production_runs`, the units and their verdicts, `inspection_runs`, and the `shipment` with its lines. One repository method, projections not tracked entities
- [ ] Include per-unit verdicts as their own entries (`kind: "verdict"`, serial + pass/scrap + reason), because "which unit failed and why" is the most interesting thing the timeline knows
- [ ] `GET /work-orders/{id}/timeline`, 404 `work_order_not_found` in the existing style. Keep the existing `/history` endpoint as-is — it is the raw state log and this is the composed narrative
- [ ] Integration test against a **fully exercised** order: a run with a forced inspection failure (`Inspection:FailureRate`) and a carrier refusal (`Shipping:RefusalRate`) so the timeline contains a rebuild, a hold, a release and a shipment — then assert it is strictly ordered and complete
- [ ] Document in the response (and in the epic) **what this endpoint is not**: it is what happened, derived from records, not the literal message log. Nothing here proves a message flowed

## Acceptance Criteria

- [ ] One request returns the full journey of an order, strictly ordered by time
- [ ] Every stage the system records appears: creation, scheduling, pick lines, each build attempt, each verdict, holds and releases, booking and dispatch
- [ ] An order that reworked and was refused a carrier shows all of it, in order
- [ ] Entry kinds are typed and stable enough for a dashboard to switch on
- [ ] 404 behaviour and problem codes match the rest of the API

## Decisions (to confirm at story start)

- **Derived from existing records, not from a persisted event log.** Settled at grooming. A log table would be truer to "event-driven", but it is a second write next to the work — precisely the dual-write problem Epic 8's transactional outbox exists to solve — and building it here would pre-empt that design with a worse version of it. **Epic 8's outbox will persist events anyway**; when it lands, an `event` kind can be merged into this same array without changing its shape. Design for that merge; do not build it.
- **No correlation id on timeline entries yet.** State history does not store one. Adding the column is cheap and would let the timeline link to the log, but it is a schema change in service of Epic 9's observability — flag it there rather than smuggling it in here.
- **Read-only and unpaginated.** A work order's timeline is bounded by its attempt cap; if it ever isn't, that is a symptom worth seeing rather than paging over.

## Notes

This story closes Epic 7 and milestone M4's happy path. After it the backend can narrate any order end to end, which is exactly the input Epic 9 (observability), Epic 10 (simulation) and Epic 11 (dashboard) all need — and the first three chapters of the portfolio write-up in Epic 16 are readable straight off this endpoint.
