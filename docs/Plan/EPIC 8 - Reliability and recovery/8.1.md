## Transactional outbox, and the work order's first concurrency token

**Labels:** messaging, backend, reliability, domain

## Summary

Close the dual-write gap that has been open since 4.1 and widened with every epic since: today a state change can commit while the event announcing it is dropped, and nothing retries. Every publisher — API *and* worker — starts writing an **outbox row in the same `SaveChanges` as the work it describes**, and a background dispatcher drains that table to RabbitMQ. While every commit path is already being touched, `WorkOrder` finally gets an optimistic-concurrency token.

## Why

This is the largest correctness hole in the system and the one the Log has been promising to fix since Epic 4. It was survivable when a dropped event only stalled picking. Since Epic 7 the publish path spans six stages, so a single swallowed publish strands an order mid-pipeline forever, with no retry and no signal — the demo's worst failure mode is silence.

It is also the enabling story for the rest of the epic. Retry, dead-lettering and replay (8.2, 8.3) are all about messages that *arrived*; none of them help a message that was never sent. Fix the send first.

## Tasks

- [ ] `OutboxMessage` entity + `outbox_messages` table: `Id` (sequential, drives ordering), `EventType` (the routing key), `Payload` (the serialized `EventEnvelope<T>`, envelope and all — including the correlation id captured **at write time**, not at send time), `OccurredUtc`, `SentUtc?`, `Attempts`, `LastError?`. Migration `Outbox`
- [ ] An `IEventPublisher` implementation that writes to the outbox instead of the broker — same interface, so no caller changes shape. The existing `RabbitMqEventPublisher` stays and becomes what the *dispatcher* uses, not what handlers use
- [ ] **The row must be enlisted in the caller's transaction**, not written in a second `SaveChanges`. This is the entire point: the outbox write and the work commit atomically or not at all. Follow 6.4's `production_runs` precedent — same unit of work, same save
- [ ] `OutboxDispatcher` (a `BackgroundService`, ~1s poll) in both hosts: claim unsent rows in id order with `FOR UPDATE SKIP LOCKED`, publish, mark `SentUtc`. Skip-locked so two instances can never publish the same row, and so a stuck row can't block the ones behind it
- [ ] Failure handling in the dispatcher: increment `Attempts`, record `LastError`, back off, leave the row unsent and try again. A broker outage must drain harmlessly once the broker returns — that is the resilience the swallow-and-log was originally protecting, kept, but now without losing the message
- [ ] Retention: mark-sent rather than delete-on-send, so a sent row is evidence for a while (Epic 11 will want it, and 7.4's timeline was explicitly shaped to merge an `event` kind later). Add a simple age-based sweep of sent rows so the table doesn't grow forever
- [ ] **`WorkOrder` gains a concurrency token** (Postgres `xmin`, mapped as a shadow property — no schema column, no domain noise). Map it in configuration the way 6.x learned read-only properties must be mapped explicitly
- [ ] Classify `DbUpdateConcurrencyException` in a handler as a **transient fault**: it is neither a business outcome nor a poison message. It nacks, and 8.2's retry ladder replays it — the reload succeeds because the conflict is gone. Until 8.2 lands it nacks-without-requeue like everything else; note the ordering dependency rather than half-building the ladder here
- [ ] Tests: a commit that fails leaves *no* outbox row; a broker that is down leaves the row unsent and it publishes when the broker returns; the dispatcher publishes in id order; two dispatchers racing the same table publish each row exactly once; a concurrent same-order advance now fails one writer loudly instead of silently last-write-wins (raced directly against Postgres in parallel scopes — the 5.3 split)
- [ ] Update [docs/messaging-topology.md](../../messaging-topology.md): publishing is now **at-least-once**, the outbox sits between the work and the exchange, and the existing dedupe keys are what make that safe

## Acceptance Criteria

- [ ] No state change can commit without its event being durably recorded for publication
- [ ] A broker outage delays events; it never loses them, and the backlog drains on recovery
- [ ] Events are published in the order they were written, per work order
- [ ] Two dispatcher instances publish each outbox row exactly once
- [ ] A concurrent update to the same work order is rejected, not silently merged
- [ ] `docs/messaging-topology.md` describes the outbox path accurately enough to redraw from

## Decisions (to confirm at story start)

- **Polled, not signalled.** A poller is restart-safe by construction and has exactly one code path; an in-process "publish right after commit, poller as fallback" has two paths to the same publish and the fast one is the one that skips on crash. ~1s of latency is invisible in a demo where stages already take longer than that.
- **All publishers, not just the worker's.** The API is where the pipeline *starts* — a dropped `work-order.scheduled` strands an order before it has moved at all. Half a fix here would leave the demo's most likely failure still possible.
- **The envelope is serialized at write time, correlation id included.** The dispatcher is a different context (a background loop, no request, no delivery) and must not invent a correlation id. The thread from 4.3 survives the outbox only if the id is captured where the work happened.
- **Publishing becomes at-least-once, deliberately.** The dispatcher can crash between `BasicPublish` and marking sent, so a row republishes. That is the correct trade — a duplicate is handled by the dedupe keys 5.4/6.4/7.1 already established, a loss is not handled by anything. Worth saying out loud in the topology doc: the outbox does not make delivery exactly-once, it converts *loss* into *duplication*, and the whole system was built to absorb duplication.
- **`xmin`, not a byte-array rowversion column.** Postgres already maintains it; adding a column would mean a migration on every table that ever needs one and a domain property that means nothing to the domain.

## Notes

Sequencing matters inside this story: get the outbox writing and draining green before touching the concurrency token, because the token will start surfacing conflicts in tests that were previously (silently) last-write-wins. Two changes, one story, but not simultaneously.

The token retires three separate "narrow, accepted for now" caveats recorded in HANDOFF — 3.4's untested concurrent advance, 6.4's manual-verdict race, and 7.2's simultaneous dispatch. Delete them from *Open decisions* when this lands; leaving them there is how a solved problem gets re-solved.
