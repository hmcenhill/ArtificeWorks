## Dead letters: inspect, and put it back

**Labels:** api, backend, reliability

## Summary

Drain `artifice.parked` into a `dead_letters` table and expose it: `GET /dead-letters` to see what failed and why, `GET /dead-letters/{id}` for the payload and the error, `POST /dead-letters/{id}/replay` to put it back into the pipeline. The first stage of the system where a human can look at a failure and un-stick it — which is what "recovery" in the epic title actually means.

## Why

8.2 stops failures from being lost, but a parked message nobody can see is only a slower kind of silence. This story turns the parked queue into something a visitor can read and act on, and it is the second half of a pattern the demo already tells well: 5.3's shortage hold and 7.3's carrier refusal both end with a human pressing release. This is the same story for infrastructure failures instead of business ones.

It is also the piece Epic 11's dashboard renders and Epic 12's failure injection targets: inject a fault, watch it retry, watch it park, click replay, watch it complete.

## Tasks

- [ ] `DeadLetter` entity + `dead_letters` table: `Id`, `EventType` (original routing key), `Payload` (the original envelope bytes, verbatim), `CorrelationId`, `WorkOrderId?` (extracted from the payload where present, so a failure can be shown next to the order it belongs to), `Attempts`, `LastError`, `ParkedUtc`, `ReplayedUtc?`, `ReplayCount`. Migration `DeadLetters`
- [ ] A `ParkedQueueDrain` hosted service in the worker: consumes `artifice.parked`, inserts the row, acks. Deliberately a *different* consumer from the main loop — the pipeline's handlers must never run for a parked message
- [ ] The drain must be the most defensive code in the system: it handles messages that are already known to be broken. A body that won't parse still gets a row (payload as raw bytes/text, `WorkOrderId` null, the parse failure as `LastError`). **A drain that throws on a poison message re-creates the exact wedge 8.2 just fixed** — prove it doesn't
- [ ] `GET /dead-letters` — paged, newest first, filterable by `workOrderId` and by replayed/unreplayed. Summary shape: id, event type, work order, attempts, parked time, first line of the error
- [ ] `GET /dead-letters/{id}` — the full record including the payload, for a human to read before deciding
- [ ] `POST /dead-letters/{id}/replay` — republish the original payload to `artifice.events` under its original routing key **via the outbox** (8.1), stamp `ReplayedUtc`, increment `ReplayCount`. Replay is a write like any other; it should not be the one place that publishes directly to the broker
- [ ] Replay resets the retry ladder — a replayed message starts at attempt 1. A human deciding to try again is a new decision, not a continuation of the automated one
- [ ] ProblemDetails codes in the established style: `dead_letter_not_found`, `dead_letter_already_replayed` (a replayed record can be replayed again, but say so deliberately — decide at story start whether a second replay is allowed or requires a `force` flag)
- [ ] Tests: a handler that always fails ends as a `dead_letters` row with its correlation id and work order intact; replay drives that work order to completion; a malformed payload becomes a readable row rather than an exception; replaying twice does not duplicate the work (the dedupe keys, again — a replayed `MaterialsReserved` for an already-picked order must ack and skip, exactly as a redelivery does)
- [ ] Update [docs/messaging-topology.md](../../messaging-topology.md): the drain, the table, and the replay path back into the exchange

## Acceptance Criteria

- [ ] Every parked message becomes an inspectable record, including ones that can't be deserialized
- [ ] The record carries enough to diagnose the failure without opening the RabbitMQ UI: what event, which order, how many attempts, what error
- [ ] A dead letter can be replayed through the API and the work completes
- [ ] Replay is idempotent against work that has since succeeded by other means
- [ ] The drain cannot be wedged by the messages it exists to handle
- [ ] Dead letters survive a broker restart or a queue purge

## Decisions (to confirm at story start)

- **A table, not a queue browser.** Browsing AMQP from a request handler is awkward, semi-destructive, and gives up the moment someone purges the queue. A table is an ordinary query, joins to the work order, survives the broker, and is what Epic 11 can render without putting AMQP in a browser.
- **Replay goes through the outbox.** Otherwise the one endpoint whose entire job is "reliably re-send this" would be the one that publishes unreliably.
- **Replay resets the ladder.** See above — a human's retry is a fresh attempt.
- **No auth yet.** `SetStatus` is still waiting on admin auth (an Open decision since Epic 3) and this endpoint belongs behind the same gate when it exists. Note it; don't build auth here. In the meantime it is *deliberately* visitor-facing — the whole demo is people pressing recovery buttons.

## Notes

Route naming: `/dead-letters` sits oddly next to `/work-orders`, which is entirely about the domain. Consider `/admin/dead-letters` or `/system/dead-letters` at story start, so the eventual auth boundary falls on a path prefix rather than a scattering of endpoints.

The summary shape here is the second thing Epic 11 renders after the timeline. Design the DTO with a list row in mind, the way 7.4's timeline was designed for a trace view.
