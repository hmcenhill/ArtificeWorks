## API idempotency keys on work order creation

**Labels:** api, backend, reliability

## Summary

Accept an `Idempotency-Key` header on `POST /work-orders`, store the key alongside a snapshot of the response, and replay that stored response when the same key arrives again. A client retry — a double-click, a flaky connection, a proxy replaying a request — produces one work order and one `201`, not two orders.

## Why

Every other reliability guarantee in this epic is about messages the system sends itself. This is the one edge the system does not control: an HTTP client that never learns whether its request landed. The rest of the pipeline is already protected — 5.4, 6.4 and 7.1's unique indexes and state machines make every downstream stage safe against repetition — and create is the last unguarded door.

It is also the smallest story in the epic by some distance, which is the point of putting it last: the epic ends on something that lands cleanly.

## Tasks

- [ ] `idempotency_keys` table: `Key` (unique), `Endpoint`, `RequestHash`, `StatusCode`, `ResponseBody`, `CreatedUtc`. Migration `IdempotencyKeys`
- [ ] Handling on `POST /work-orders`:
  - no header → behave exactly as today. The header is opt-in; a demo visitor with curl should not need to know it exists
  - unseen key → process normally, then store key + request hash + the response actually returned, in the **same transaction as the work order** (5.4's rule again: the marker commits with the work, or a crash between them makes the guarantee a lie)
  - seen key, same request hash → replay the stored status and body verbatim, no new order
  - seen key, **different** request hash → `422` with code `idempotency_key_reused`. Same key, different body is a client bug, and silently replaying the first response would hide it
- [ ] The unique constraint is the guarantee, not the pre-check read: two simultaneous requests with the same key must produce one order. Catch the constraint violation and replay the winner's response — the same check-then-act shape 5.4 established
- [ ] Only successful responses are stored. A `400` should not be permanently pinned to a key the client will sensibly retry with a corrected body
- [ ] Age-based cleanup of old keys, alongside 8.1's outbox sweep
- [ ] Swagger: document the header, and say plainly that it is optional
- [ ] Tests: same key twice → one order, identical response both times; same key with a different body → `422 idempotency_key_reused`; two concurrent requests with one key → one order (raced against Postgres); no key → unchanged behaviour, proven by the existing suite still passing untouched

## Acceptance Criteria

- [ ] A repeated create with the same idempotency key creates exactly one work order
- [ ] The replayed response is byte-identical to the original, including the created order's id
- [ ] Concurrent duplicate requests are resolved by a database constraint, not a read
- [ ] Reusing a key with a different body is rejected with a stable ProblemDetails code
- [ ] Requests without the header are entirely unaffected

## Decisions (to confirm at story start)

- **Create only.** The other write endpoints are already idempotent by construction and 7.x proved it: `advance`/`hold`/`release` are state-machine transitions that reject a repeat, `POST /inspections` has a per-unit guard, `POST /shipments` has a unique index. Adding keys there would buy a nicer error message for a write per request — a bad trade. If a second endpoint ever genuinely needs one, the table and the filter are already here.
- **Filter or middleware, not handler code.** The mechanism must be indifferent to which endpoint it wraps, so extending it later is registration and nothing else.
- **Store the response, not just the key.** A bare "already seen" marker would have to answer the retry with a `409`, and a client that never received the first response has no way to learn the order's id. Replaying the original `201` is the only answer that leaves the client correct.
- **`422`, not `409`, for a reused key with a different body.** `409` is already the duplicate-product code from 3.3; the situations are different and the codes should be too. Confirm against the existing ProblemDetails table at story start rather than guessing.

## Notes

The whole epic's throughline lands here, and it is worth writing into the portfolio notes: *the key follows the thing that must happen once*. 5.4 said the order, 6.4 said the attempt, 7.1 said the order again, and here it is the client's request. Four stages, one rule, four different keys — that is the reliability argument the demo is making.
