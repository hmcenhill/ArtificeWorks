## Metrics: the factory's vital signs, and a feed for the dashboard

**Labels:** observability, backend

## Summary

One `Meter` for the pipeline, instrumented at the points that already exist — stage transitions,
builds, verdicts, bookings, retries, parks, outbox rows — plus observable gauges for the queue-ish
things (outbox lag, unreplayed dead letters, orders per status). Exported over OTLP to Prometheus,
and mirrored by a small `GET /system/stats` JSON snapshot for Epic 11.

## Why

Epic 8 deliberately left a lot to count: retry counts, parked messages, outbox lag, the
`dead_letters` table. Right now the only way to know any of it is to query Postgres by hand or
squint at the RabbitMQ management UI. Metrics turn "the pipeline is reliable" from a claim in a
document into a graph that moves when you break something — which is precisely the demo Epic 12
is building toward.

The snapshot endpoint exists because the dashboard should not have to speak PromQL, embed Grafana
auth, or care whether a metrics backend is running at all.

## The shape of it

Three families, and they are not interchangeable:

| Family | Instrument | Examples |
|---|---|---|
| **Things that happened** | `Counter` | orders created, stage transitions (tagged `from`/`to`), units built / passed / scrapped, rework attempts, shipments booked / dispatched / refused, retries (tagged by rung), parks, replays, outbox rows published |
| **Things that took time** | `Histogram` | handler duration by event type, publish duration, time an order spent in each stage |
| **Things that are true right now** | `ObservableGauge` | unsent outbox rows, age of the oldest unsent row (**the lag**), unreplayed dead letters, work orders per status |

The gauges are the ones with a trap in them: their callbacks run on the meter's collection thread,
in root scope, and each wants a database query. Naively that is a `SELECT` per metric per scrape,
from a callback that cannot resolve a scoped `DbContext`.

## Tasks

- [ ] `ArtificeWorksMetrics` in Infrastructure — a single `Meter` named `ArtificeWorks.Pipeline`,
      registered as a singleton, with every instrument declared in one file. One file is the point:
      metric names are a public contract with Grafana and with Epic 11, and they should be
      reviewable in one screen. Follow OTel naming (`artificeworks.work_orders.transitions`,
      `artificeworks.messages.retried`, dots not underscores, units on the instrument)
- [ ] Instrument at the existing seams, not by sprinkling: the four workflow services
      (`MaterialPickingService`, `ProductionService`, `InspectionService`, `ShippingService`), the
      consumer loop's three-way classification (8.2), `OutboxDispatcher`, and `ParkedQueueDrain`.
      A stage transition should be counted in exactly one place
- [ ] Observable gauges backed by a **short-lived cached snapshot** (a hosted service refreshing
      every few seconds into a singleton), not by a query inside the callback. Also solves the
      scoped-`DbContext`-from-root-scope problem, and keeps a Grafana scrape from being able to
      load the database
- [ ] Turn on the built-in meters both hosts already emit for free: ASP.NET Core, HttpClient,
      `Microsoft.EntityFrameworkCore`, `System.Runtime` (GC, threadpool). These cost one line each
      and are half of what a real dashboard shows
- [ ] Queue depth: prefer RabbitMQ's own metrics over inventing our own. Record what is available
      from the client and where the rest comes from (the management API), and **don't build a
      poller for it in this story** — the parked/dead-letter counts, which are rows in our
      database, are the ones the demo actually needs
- [ ] `GET /system/stats` — one JSON snapshot: counts per stage, in-flight orders, outbox unsent +
      lag seconds, unreplayed dead letters, retry/park counts since start. Under `/system` so it
      sits behind the same eventual admin gate as `/system/dead-letters` (8.3). Served from the
      same cached snapshot the gauges use, so the endpoint and Grafana can never disagree
- [ ] A provisioned Grafana dashboard checked into the repo (`docker/grafana/`) so
      `docker compose up -d` yields a factory-floor view with no clicking. This is a demo asset, not
      a nicety — M7 embeds it
- [ ] Tests: a work order driven end to end moves the transition counters by exactly one per stage
      (the check that catches double-counting); a forced handler failure increments the retry
      counter and then the park counter; `GET /system/stats` reflects seeded state. Use
      `MetricCollector<T>` from `Microsoft.Extensions.Diagnostics.Testing` rather than asserting
      against a scrape

## Acceptance Criteria

- [ ] Throughput, failure, retry and park counts are exposed per pipeline stage
- [ ] Outbox lag and unreplayed dead-letter count are visible as gauges
- [ ] Metrics reach Prometheus via the 9.1 OTLP endpoint and render in a checked-in dashboard
- [ ] `GET /system/stats` returns the same numbers without a metrics backend running
- [ ] No metric collection issues a database query per scrape
- [ ] Metric names are declared in exactly one file

## Decisions (to confirm at story start)

- **Both a meter and an endpoint** (settled at grooming). Instrument once with
  `System.Diagnostics.Metrics`; expose a cheap JSON mirror for the React dashboard. The
  alternative — Epic 11 querying Prometheus — pushes a query language, a CORS problem and an auth
  problem into the frontend epic to save an endpoint that is a few dozen lines over data that
  already exists.
- **`System.Diagnostics.Metrics`, not a Prometheus client library.** Vendor-neutral, first-class in
  .NET, and the same API the framework's own meters use — so ours and theirs export through one
  pipeline.
- **Counters at the workflow services, not in the domain.** `WorkOrder` depends on nothing and
  should keep depending on nothing. The service that commits the transition is the honest place to
  count it, and it is also where the correlation id and the trace already are.
- **No per-work-order labels. Ever.** Order id, correlation id and trace id are unbounded
  cardinality and would ruin the metrics backend. Those questions belong to traces and logs, which
  is exactly the division of labour this epic is arguing for.

## Notes

Independent of 9.1 for the instrumentation itself; depends on it only for the exporter wiring, and
falls back to the snapshot endpoint if that slips.

The gauge-caching hosted service is a third background loop next to `OutboxDispatcher` and
`RetentionSweepService`. Consider whether they should share one timer host before adding a fourth
in a later epic.
