## Logs: structured, correlated, and worth reading

**Labels:** observability, backend

## Summary

Take the logs from "console lines with a correlation id prefix" to a queryable structured stream:
OTLP export into Loki alongside the console, trace and span ids on every line automatically, the
correlation id joined to the trace via 9.1's baggage, and a levelling pass over the whole system so
what's emitted at Information is what a person actually wants during a demo.

## Why

The logs are the oldest observability surface here — 4.3 built them — and the only one that has
never been revisited while five epics were bolted on. They are also the surface Epic 12's audience
will be *watching* while they inject failures, so the signal-to-noise ratio stops being a
developer-comfort question and becomes a demo question.

And a structured log is what closes the loop the other two signals leave open: a trace shows you
that the pick span failed; the log line inside it tells you it was component `HAW-SVO-002` and
there were four on the shelf.

## The shape of it

```
correlation id ──┬── log lines   (grep / LogQL: {correlation_id="b1f3…"})
                 └── spans       (baggage → span attribute)
trace id ────────┬── spans       (the waterfall)
                 └── log lines   (auto-attached by the OTel logging provider)
```

Two ids, four directions, no dead ends: from any line you can reach the trace, and from any span
you can reach the lines.

## Tasks

- [x] Add the OpenTelemetry logging provider to both hosts (through 9.1's shared
      `AddArtificeWorksTelemetry`, not separately), with `IncludeScopes` and
      `IncludeFormattedMessage` on so scope values arrive as structured attributes rather than
      being flattened into the message. Trace and span ids are attached by the provider — no code
      in handlers
- [x] **Keep the console.** OTLP is for querying; the console is what a person watches during a
      demo and what `docker logs` shows when the collector is the thing that's broken. Console
      stays human-readable in Development; JSON console in other environments
- [x] Promote the correlation id from a message-template scope to a first-class structured field.
      `CorrelationLog` already exists as the single place the field name is defined — keep that
      property and extend it rather than adding a second mechanism, and confirm the emitted
      attribute name matches the span attribute 9.1 sets
- [x] Resolve the **`IncludeScopes` appsettings-vs-code footgun** flagged in the 4.3 log and open
      ever since. Settle where scope rendering is configured and make both hosts agree, in one
      place, with a comment saying why
- [x] A levelling pass across the system, judged by "what does a visitor watching a demo need to
      see?":
  - every stage transition, exactly once, at Information with the work order id
  - retries, parks and holds at Warning; faults and unexpected failures at Error
  - the outbox dispatcher's per-pass chatter, EF Core command logging and the AMQP client's
    connection noise down to Debug — these are the three things currently drowning the interesting
    lines
  - no `Exception.ToString()` pasted into a message string; exceptions go in the exception
    parameter where the backend can index them
- [x] Audit for **log lines that should exist and don't**, which the levelling pass will expose:
      the most likely gaps are the stage transitions the workflow services perform silently and
      anything that swallows an exception
- [x] Loki/LogQL: confirm the fields actually land as labels/fields the way the queries in 9.4's
      runbook assume. A structured field that only exists inside the rendered message is not
      queryable, and that is the failure mode to test for
- [x] Tests: an integration test asserts a log record emitted during a request carries both the
      correlation id field and a non-zero trace id; a worker-side test asserts the same for a
      delivery, with the trace id matching the publisher's. Use `FakeLogCollector` from
      `Microsoft.Extensions.Diagnostics.Testing`

## Acceptance Criteria

- [x] Correlation IDs are logged consistently across both services, as structured fields
- [x] Every log line emitted inside an activity carries its trace and span ids
- [x] A correlation id can be pivoted to a trace, and a trace back to its lines
- [x] Logs are queryable in Grafana without the console output getting worse
- [x] Information-level output for one work order's journey is short enough to read live

## Decisions (to confirm at story start)

- **Microsoft.Extensions.Logging with the OTel provider, not Serilog.** The codebase is `ILogger`
  throughout and every scope already goes through `CorrelationLog`; Serilog would buy sinks we
  don't need and a second configuration surface. The OTel provider gives structure, trace
  correlation and OTLP export with no call-site changes.
- **Console and OTLP both, always.** Not either/or. The demo watches one and the runbook queries
  the other, and the console is the fallback when the observability stack itself is what failed.
- **The levelling pass is part of this story, not a follow-up.** Structured logging that nobody can
  read is the same problem in a nicer format, and the noise is measurable right now: an idle worker
  logs an outbox pass every second.

## Notes

Depends on 9.1 for trace ids and for the shared telemetry registration; standalone otherwise.

There is a real risk of this story sprawling into a rewrite of every log statement in the system.
The levelling pass is scoped to *levels, structure and the named gaps* — not to rewording messages
that are merely inelegant.
