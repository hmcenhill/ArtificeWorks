## Health, readiness, and "what happened to work order X?"

**Labels:** observability, backend, docs

## Summary

Turn `/health` — one line that has returned "Healthy" unconditionally since Epic 1 — into real
liveness and readiness probes over the dependencies the pipeline actually has, and write
`docs/observability.md`: the runbook that says which of the four surfaces answers which question,
and how to bring the whole stack up locally.

## Why

Two loose ends, and they belong together because both answer "is this thing working, and how would
I know?".

`AddHealthChecks()` with no checks registered is worse than nothing: it reports Healthy while
Postgres is down, and M7 will point a container orchestrator at it. Epic 15 needs probes that
mean something, and the checks are cheap once the dependencies are all in the container.

The runbook is an explicit acceptance criterion of this epic and is currently owned by no story.
By the time 9.1–9.3 land there will be **four** ways to answer a question about a work order — the
timeline endpoint (7.4), the logs, the traces, the metrics — and four overlapping tools with no
guidance is its own kind of unobservable.

## Tasks

- [x] Split the endpoint by purpose, because they answer different questions and a wrong answer
      either way is an outage:
  - `GET /health/live` — is the process up? No dependency checks at all. A failing liveness probe
    means *restart me*, and restarting the API does not fix a dead database
  - `GET /health/ready` — can it serve? Postgres reachable, RabbitMQ connection open, no pending
    migrations
  - keep `GET /health` as an alias for readiness so nothing that already points at it breaks
- [x] Checks, each tagged so the endpoints can filter by tag rather than by name:
      Postgres (`SELECT 1` through the existing `DbContext`, not a new connection), RabbitMQ
      (connection open — the connection object already exists, so this must not open a second one),
      pending migrations (the API already computes this at startup for the seeder; reuse it), and
      **outbox lag as Degraded, never Unhealthy** — a backlog means the broker is struggling, and
      taking the API out of rotation would stop new work from being recorded while doing nothing
      to drain it
- [x] The worker host needs a health endpoint too — it has none, and it is the half more likely to
      be wedged. A minimal HTTP listener or the same health-check services over a tiny endpoint;
      keep it deliberately small
- [x] Health check results should be a JSON response with per-check status and duration, not the
      bare string. This is the thing an operator reads first
- [x] **`docs/observability.md`** — the epic's deliverable, written for someone who has never seen
      the system:
  - the four surfaces and the question each one owns: *what happened to this order* → the timeline
    endpoint; *why did it fail* → logs by correlation id; *where did the time go* → the trace;
    *is the factory healthy* → metrics and `/health/ready`
  - **"What happened to work order X?", worked end to end** as a single narrative: start from an
    order id, get its correlation id, pivot to the trace, pivot to the log lines, read the
    timeline. Copy-pasteable LogQL and PromQL, and the actual Grafana URLs the compose file yields
  - local setup: `docker compose up -d`, what each container is for, which ports, and how to
    confirm telemetry is flowing
  - a short "if you see X, look at Y" table for the failure modes Epic 8 built — outbox lag
    climbing, messages parking, an order stuck in one stage
- [x] Cross-link it: `README.md`, `CLAUDE.md`'s commands section, and a pointer from
      `docs/messaging-topology.md` (which currently owns the correlation story and should hand off
      rather than duplicate)
- [x] Tests: readiness reports Unhealthy when the database is unreachable and Healthy when it is;
      liveness stays Healthy regardless; outbox backlog produces Degraded rather than Unhealthy;
      the response body has the per-check shape the docs promise

## Acceptance Criteria

- [x] `/health/live` and `/health/ready` exist on the API and mean different things
- [x] Readiness fails when Postgres or RabbitMQ is unavailable, and when migrations are pending
- [x] Outbox backlog degrades readiness without failing it
- [x] The worker host exposes a health signal
- [x] `docs/observability.md` answers "what happened to work order X?" end to end with real,
      runnable queries
- [x] Local setup for viewing logs, metrics and traces is documented and works from a clean clone

## Decisions (to confirm at story start)

- **Liveness and readiness are separate, and liveness checks nothing.** The classic mistake is one
  endpoint checking dependencies: the database blips, every replica reports unhealthy, the
  orchestrator restarts all of them, and the restarts make it worse.
- **Outbox lag is Degraded, not Unhealthy.** It is a symptom of a downstream problem that removing
  this instance cannot fix — and 8.1's whole design is that a backlog is *safe*, because nothing is
  lost while it drains.
- **The runbook is one document, not notes spread across the story files.** Story files are
  planning artifacts and go stale by design; `docs/` is where things live once they're true. Same
  rule that put the topology in `docs/messaging-topology.md`.
- **Admin auth is still not in scope.** `/health/*` is meant to be unauthenticated; `/system/stats`
  and `/system/dead-letters` continue to wait for the gate that has been deferred since 8.3, which
  is why they stay under one path prefix.

## Notes

Health checks are independent of the rest of the epic and could be built first. The runbook
genuinely cannot be written until 9.1–9.3 have landed, since it documents them — so if the epic
has to stop early, this story splits cleanly along that line.

Care with the RabbitMQ check: an aggressive readiness probe that opens a connection per call is a
real way to exhaust a broker. Reuse `IRabbitMqConnection`.
